# Prompt 收口与 `agent.md` 迁移规划

## 1. 本轮目标

本轮不再把重点放在“多份 prompt 并存”，而是收口为：

- **只保留一份 `FULL` 一键提示词** 作为唯一主入口
- 其他 prompt 版本逐步退出，不再作为正式接入路径
- 协议最终不只停留在文档里，而是要迁移到真正会被主 Agent 读取的 `agent.md / AGENTS.md` 中

一句话：

**先在文档里把主 Agent 运行协议规划清楚，再把它压缩、迁移、固化进 `agent.md`。**

---

## 2. 为什么要只保留 `FULL`

当前仓里同时存在：
- `prompt/PROMPT_FULL.md`
- `prompt/PROMPT_SHORT.md`
- `prompt/modular-prompt.md`
- `prompt/one-shot-prompt.md`

这会带来几个问题：

### 2.1 入口分裂
用户或接入方会犹豫：
- 到底用哪一份
- 哪一份才是当前真值
- 哪一份会持续维护

### 2.2 规则漂移
当主 Agent 运行协议升级时，多份 prompt 很容易出现不同步。

### 2.3 执行口径不稳定
你现在真正需要的是：
- 主 Agent 收到任务后稳定按同一套协议行动
- 而不是不同入口带来不同行为

所以本项目后续应收口为：

- `FULL` = 唯一正式 prompt 真值
- 其他 prompt = 历史过渡产物，应逐步移除或降级为归档参考

---

## 3. 新的层级关系

后续建议把整个项目分成三层：

## 3.1 规划层（docs）
作用：
- 先把主 Agent 运行协议、拆解协议、分发协议、汇总协议写清楚
- 便于讨论、审查、重构
- 便于从“概念规则”过渡到“可执行规则”

注意：
- docs 是规划真值和解释层
- 不是最终执行入口

---

## 3.2 提示层（`prompt/PROMPT_FULL.md`）
作用：
- 把规划层压缩成可一键部署的主 Agent 提示词
- 让主 Agent 真能按这套协议工作

要求：
- 只保留一个正式版本
- 可直接贴到 system / boot / persona prompt 中
- 必须包含主 Agent 的判断、拆解、分发、协调、回收、裁决、放行闭环

---

## 3.3 执行层（`agent.md / AGENTS.md`）
作用：
- 作为最终长期运行入口
- 让主 Agent 在真实环境中稳定读取和遵循规则
- 把最重要的主控协议从“提示词临时注入”变成“工作区长期规则”

要求：
- 不只是角色介绍
- 必须承载运行协议
- 应优先写“收到任务后怎么动作”，而不是只写“谁是谁”

---

## 4. 真正要迁移进去的内容是什么

迁移进 `agent.md / AGENTS.md` 的，不应只是角色定义。

真正应该迁移的是：

### 4.1 主 Agent 权限边界
- 什么能直接做
- 什么必须拆
- 什么必须复核
- 什么不能下放
- 什么必须由主 Agent 最终裁决

### 4.2 主 Agent 默认动作顺序
收到任务后，按什么顺序动作：
- 判断目标清晰度
- 判断复杂度等级
- 判断风险
- 判断是否直做
- 判断是否拆解
- 判断是否需要协调层
- 判断是否并行
- 回收结果
- 汇总裁决
- 最终放行

### 4.3 任务拆解协议
- 什么情况下拆
- 拆成几个子任务
- 子任务边界如何写
- 需要什么输入
- 期望什么输出
- 如何定义完成标准

### 4.4 分发协议
- 派给谁
- 为什么派给他
- 要求回报什么
- 是否允许并行
- 是否要求 review / handoff

### 4.5 协作协议
- 多 agent 串行 / 并行条件
- 承枢何时介入
- 结果冲突谁裁决
- 失败 / 阻塞 / 超时怎么处理

### 4.6 汇总协议
- 各 agent 的汇报必须包含什么
- 主 Agent 如何去重、去噪、识别冲突
- 如何输出最终结论
- 如何给出风险和未完成项

---

## 5. docs 先规划什么

在正式迁移前，先在 docs 层补齐以下几个规划文件：

### A. 主 Agent 运行协议
建议以 `docs/main-agent-decision-flow.md` 为核心重写，重点从“原则说明”升级为“执行流程”。

### B. 分发 / 派单协议
建议以 `docs/handoff-protocol.md` 或新增专项文档承接，重点明确：
- 主 Agent 派单格式
- specialist 回传格式

### C. 汇总 / 裁决协议
建议补明确：
- 多 agent 结果如何回收
- 冲突如何判定
- 主 Agent 最终总结结构是什么

### D. 高风险与放行协议
明确：
- 哪些只能出方案稿
- 哪些可以直接执行
- 哪些必须人工确认

---

## 6. `FULL` 的目标状态

最终 `PROMPT_FULL.md` 不应该只是“规则汇总”，而应该是：

**一份主 Agent 可直接执行的最小运行协议。**

至少应包含：
- 主 Agent 身份与边界
- 默认动作顺序
- L1-L4 分级
- 拆解条件
- 分发格式要求
- 协调与并行规则
- handoff / review / release 规则
- 结果回收与汇总规则
- 高风险默认动作
- 真实多 agent 调度与模拟调度的处理方式

---

## 7. `agent.md / AGENTS.md` 的目标状态

最终 `agent.md / AGENTS.md` 不是文档目录，也不是角色介绍页。

它要变成：

**主 Agent 的长期运行宪章。**

应该优先写：
1. 收到任务如何判断
2. 什么情况下拆解
3. 怎么分配
4. 怎么协调
5. 怎么回收
6. 怎么总结
7. 哪些绝不能越权

角色介绍、风格、口吻这些内容可以保留，
但不能压过“运行协议”本体。

---

## 8. 建议的推进顺序

推荐按以下顺序推进：

### 第一步
先重写 docs 中最核心的运行协议文件，确保规则逻辑稳定。

### 第二步
把 docs 中的核心协议压缩进 `prompt/PROMPT_FULL.md`，形成唯一一份正式 prompt。

### 第三步
清理或降级其余 prompt 文件：
- `PROMPT_SHORT`
- `modular-prompt`
- `one-shot-prompt`

### 第四步
把最终稳定版本迁移进工作区实际入口文件：
- `workspace-template/AGENTS.md`
- 如有必要，再补 `team/*/AGENTS.md` 或 `agent.md`

### 第五步
更新 `README.md`、`QUICKSTART.md`、`INSTALL.md`，统一口径：
- 只保留 `FULL`
- docs 用于规划
- `AGENTS.md` 用于长期运行

---

## 9. 本轮收口结论

本轮方向应正式改为：

### 结论 1
本项目 prompt 入口应收口为：

**只保留 `prompt/PROMPT_FULL.md` 作为唯一正式一键部署入口。**

### 结论 2
其他 prompt 文件不再作为长期正式入口，应逐步退出主路径。

### 结论 3
真正关键的协议应最终进入：

**`agent.md / AGENTS.md`**

因为只有进入主 Agent 真会读取的入口文件，规则才会真正生效。

### 结论 4
在迁移前，可以先在 docs 中完成规划和重构，避免直接在运行入口里反复试错。
