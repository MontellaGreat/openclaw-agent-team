# Agent I/O Contracts

本文件定义各角色在真实协作中的最小输入输出契约。

目标：
- 让“谁做什么”升级成“收到什么、交出什么”
- 让墨影能更快回收结果并裁决
- 让交接从口头描述升级为可复用的结构化协议

---

## 一、为什么需要 I/O 契约

如果只有角色说明，没有输入输出契约，就会出现：
- 墨影发任务时边界不清
- 附属 agent 交回结果时格式不统一
- 下游 agent 接不上
- 控制台和复盘难以结构化记录

所以必须给每个角色定义：
- 输入最小要求
- 输出最小要求
- 常见失败条件
- 何时必须升级给墨影

---

## 二、统一输入模板

墨影分发任务时，建议至少提供：

```text
1. 任务目标
2. 任务边界
3. 当前上下文
4. 需要的输出格式
5. 风险提示
6. 是否需要主 Agent 裁决
```

若缺少这些信息，附属 agent 应先补问题，而不是直接盲做。

---

## 三、统一输出模板

所有附属 agent 交付时，至少包含：

```text
1. 做了什么
2. 结论 / 产出
3. 产物位置
4. 验证方式
5. 风险 / 不确定点
6. 是否建议放行
7. 下一步建议
```

---

## 四、角色级契约

### 1. 探针

#### 输入
- 要研究的主题
- 研究边界
- 是否需要比较方案
- 是否需要给推荐路径

#### 输出
- 结论
- 证据要点
- 可信度判断（确定 / 高概率 / 待验证）
- 推荐路径
- 不确定点

#### 不应输出
- 不带证据的强判断
- 把未验证内容写成定论

#### 必须升级给墨影
- 方案冲突明显
- 证据不足却影响路线选择
- 需要速度/质量/风险取舍

---

### 2. 笔官

#### 输入
- 文稿目标
- 面向对象
- 事实材料
- 风格要求
- 是否用于正式对外发布

#### 输出
- 成稿或提纲
- 当前版本定位（草稿 / 可发布 / 待补证据）
- 尚未确认的事实点
- 是否建议对外使用

#### 不应输出
- 无事实基础的成稿
- 把未定内容假装已定稿

#### 必须升级给墨影
- 对外正式发布
- 事实仍有不确定项
- 用户目标与现有材料冲突

---

### 3. 观象

#### 输入
- 图片 / 页面 / 截图 / OCR 对象
- 用户想看什么
- 是否只提事实，还是要带解释

#### 输出
- 视觉事实
- OCR 文本
- 明确标注哪些是推测
- 若有约束，指出识别局限

#### 不应输出
- 把推测写成事实
- 跳过画面限制直接下结论

#### 必须升级给墨影
- 结论会影响后续路线判断
- 用户要求基于截图做高风险决策

---

### 4. 铁手

#### 输入
- 任务目标
- 代码边界
- 是否允许改配置 / 服务 / 权限
- 是否需要测试或最小验证

#### 输出
- 改了什么
- 没改什么
- 产物位置
- 如何验证
- 风险点
- 回滚点（如适用）

#### 不应输出
- 改动不交代范围
- 改了高风险项却不提示
- 没验证却包装成可放行

#### 必须升级给墨影
- 配置变更
- 权限 / 密钥 / 服务修改
- 删除覆盖
- 部署发布
- 风险未知但影响大

---

### 5. 问隙

#### 输入
- 待验证对象
- 验证目标
- 验证边界
- 是否要给放行建议

#### 输出
- 覆盖范围
- 未覆盖范围
- 风险等级
- 问题列表
- 是否建议放行

#### 不应输出
- 只说“看起来没问题”
- 不说明未覆盖范围

#### 必须升级给墨影
- 存在放行争议
- 高风险项未能验证
- 验证结果与铁手结论冲突

---

### 6. 片场

#### 输入
- 素材目标
- 素材类型
- 风格要求
- 交付级别（方向稿 / 草案 / 正式版）

#### 输出
- 任务类型判断
- 已调用的子 agent
- 当前产物级别
- 产物位置
- 下一轮迭代建议

#### 不应输出
- 目标没澄清就直接开工
- 草案假装正式版

#### 必须升级给墨影
- 对外正式使用
- 高成本生成
- 方向不确定

---

## 五、建议的机器可读结果卡

建议所有附属 agent 输出时，尽量兼容以下字段：

```json
{
  "agent": "tieshou",
  "task_id": "task-20260325-001",
  "result_summary": "一句话结论",
  "artifacts": [
    {
      "type": "code",
      "path": "src/example.ts",
      "desc": "改动文件"
    }
  ],
  "verification": "如何验证",
  "risk_level": "medium",
  "release_suggestion": "review_required",
  "next_step": "建议下一步"
}
```

---

## 六、一句话原则

> 角色说明解决“谁来做”；I/O 契约解决“怎么接、怎么交、怎么判”。
