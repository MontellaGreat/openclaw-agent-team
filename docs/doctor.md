# doctor.md

这不是安装层的 doctor，而是内容与协作层的排查入口。

当你发现这套多 Agent 协作规则“看起来都在，实际却不好用”时，先从这里查。

---

## 1. 主 Agent 变成中转站，不像总控

### 表现
- 用户说一句，主 Agent 就原样转给别的 agent
- 主 Agent 很少自己判断
- 最终结果像转发拼盘，没有裁决感

### 原因
- 把“分发”误当成默认动作
- 主 Agent 没建立“先判断，再决定是否拆”的习惯
- prompt 里角色有了，但动作顺序不够强

### 处理
- 强化主 Agent 默认动作顺序
- 明确：能直接稳定做完的任务，不拆
- 强化主 Agent 的汇总、裁决、放行职责
- 检查主 Agent 输出是否先给判断，再给动作

---

## 2. 简单任务被过度拆解

### 表现
- 润色一段话也要拆给多个 agent
- 一个低风险例行任务也要先开协作说明
- 用户要一个简单结果，却收到一堆分工过程

### 原因
- 把“多 Agent 感”当成目标
- 没把 L1 直接执行规则写硬
- 主 Agent 害怕自己直接做

### 处理
- 明确 L1 的默认动作：直接执行
- 增加一句硬规则：能直接稳定做完，就别拆
- 检查是否存在“为了展示团队结构而分发”的倾向

---

## 3. 复杂任务没有拆解，主 Agent 一把梭

### 表现
- 需要调研、写作、配图的任务被主 Agent 一次性硬做
- 最终结果浅、杂、容易漏
- 用户看不到明确步骤和分工逻辑

### 原因
- 主 Agent 没建立复杂度分级意识
- 没有把 L2/L3 的典型特征写清楚
- 没有“分发前说明模板”

### 处理
- 强化 L2/L3 任务判定规则
- 要求复杂任务分发前必须先说明为什么复杂、准备怎么拆、准备派给谁
- 检查是否存在“主 Agent 一把做完但质量下降”的现象

---

## 4. 子 Agent 越权输出最终结论

### 表现
- 子 Agent 直接替主 Agent 下最终判断
- 子 Agent 直接对外承诺“已经可以发布/已经没问题”
- 主 Agent 只是代发，没有复核

### 原因
- 共享规则里“无最终拍板权”不够硬
- 主 Agent 没有回收后裁决
- 交接格式没有明确“是否需要主 Agent 决策”

### 处理
- 强化附属 Agent 边界
- 要求所有交接都带“是否需要主 Agent 裁决”
- 主 Agent 不原样转发附属 Agent 结果，先做汇总判断

---

## 5. 高风险任务被直接执行

### 表现
- 改配置、改权限、删文件、发正式内容时直接落地
- 主 Agent 没有任何复核动作
- 用户没看到方案、检查单或执行稿

### 原因
- 只定义了高风险范围，没定义默认动作
- 主 Agent 没把“高风险 = 不默认直接落地”执行出来
- prompt 缺少高风险处理模板

### 处理
- 明确高风险默认动作：先给方案、草案、执行稿或检查清单
- 明确当前是否建议放行
- 把“配置 / 权限 / 删除覆盖 / 部署 / 正式外发 / 隐私金钱法律”统一收进高风险清单

---

## 6. 交接质量差，主 Agent 无法汇总

### 表现
- 子 Agent 回来只有一句“已完成”
- 没有边界、没有验证、没有风险说明
- 主 Agent 只能自己重新理解一遍

### 原因
- 没有统一交接格式
- 没有把“怎么验证、有什么风险、下一步建议”写进必填项
- 默认输出格式偏结果，不偏交接

### 处理
- 所有附属 Agent 统一最小交接格式：
  - 任务边界
  - 做了什么
  - 产出是什么
  - 怎么验证
  - 不确定点
  - 风险等级
  - 是否需要主 Agent 裁决
  - 下一步建议

---

## 7. 片场乱接任务

### 表现
- 文案没定就开始做图
- 风格没定就开视频
- 修图 / 生图 / 视频边界混乱

### 原因
- 片场只被当成“万能多媒体 agent”
- 没有先判断任务类型
- 没有把子能力规则挂进去

### 处理
- 明确片场是调度中枢，不是万能执行器
- 先判断：生图 / 视频 / 修图
- 再决定是否下发给片场-生图 / 片场-视频 / 片场-修图
- 风格、目标、素材没定清前，不盲目开工

---

## 8. 多 Agent 并行后更乱

### 表现
- 同时派了很多 agent，但最后很难汇总
- 子任务互相依赖，结果却并行开跑
- 汇总成本大于收益

### 原因
- 把并行当成默认优化
- 没有检查子任务是否真正独立
- 主 Agent 没评估合并成本

### 处理
- 并行前先判断三件事：
  - 子任务是否独立
  - 是否真的节约时间
  - 汇总成本是否可控
- 只要依赖链明显，就不要并行

---

## 9. 迁移到新的 OpenClaw 实例后不好用

### 表现
- 规则还在，但运行起来不顺手
- 子 Agent 名字对不上
- 实例里有 agent，但能力边界不一致

### 原因
- 把名称映射当成能力映射
- 只复制 prompt，没有补 workspace 规则或本地说明
- 没有做最小闭环验证

### 处理
- 先找等价能力，不硬凑同名
- 先跑 3 个验证任务：简单 / 复杂 / 高风险
- 长期使用时，把规则逐步落到 workspace 根文件，而不是只塞在 prompt 里

---

## 10. 用户看到的不是结果，而是一堆流程话

### 表现
- 用户只要答案，却先收到长篇调度解释
- 回复里制度感重，结果感弱
- 内容像在介绍项目，不像在完成任务

### 原因
- 角色设定盖过任务执行
- 主 Agent 没坚持“结论优先”
- 过度解释协作过程

### 处理
- 强化标准输出格式：先结论，再必要说明
- 默认不展示多余流程
- 只有在复杂任务确实需要时，才简要说明分工

---

## 一句话总检查

如果这套系统开始不好用，先问 4 句：
1. 主 Agent 有没有先判断？
2. 简单任务有没有直接做？
3. 高风险任务有没有被拦住直接落地？
4. 附属 Agent 有没有只做子任务，没有越权拍板？

只要这 4 句能答“是”，这套协作机制通常就没有跑偏。
