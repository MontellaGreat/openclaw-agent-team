# Failure Recovery

本文件定义多 Agent 运行中的失败、阻塞、冲突与降级策略。

目标：
- 让失败成为可处理状态，而不是现场混乱
- 让墨影知道什么时候该重试、什么时候该降级、什么时候该终止
- 让多 Agent 协作具备最小恢复能力

---

## 一、为什么需要失败恢复协议

多 Agent 系统最容易出问题的地方，不是不会分工，而是：
- 子 agent 超时
- 子 agent 没结果
- 多个结果互相冲突
- 工具失败导致链路中断
- 一部分做完、一部分失败
- 高风险任务推进到一半才发现不该放行

如果没有统一恢复协议，结果就会变成：
- 墨影临场救火
- 状态失真
- 用户看到的是混乱过程，而不是清楚结论

---

## 二、失败类型

### 1. `timeout`
子 agent 在合理时间内没有返回结果。

### 2. `tool_failure`
工具调用失败，导致任务无法继续。

### 3. `blocked_dependency`
缺少权限、文件、账号、环境、前置信息。

### 4. `conflicting_results`
多个附属 agent 的结果互相冲突。

### 5. `partial_success`
部分子任务完成，部分失败。

### 6. `risk_escalation`
执行过程中发现风险等级高于预期。

### 7. `invalid_handoff`
有交接，但不足以让下游接手或不足以让墨影裁决。

---

## 三、恢复优先级

墨影遇到失败时，按以下顺序判断：

1. 这是不是临时性问题
2. 是否值得重试一次
3. 是否该换路径
4. 是否应该降级为主 Agent 直接收口
5. 是否必须进入 `blocked` / `stopped`

原则：
> 不为了“看起来完成”而强行推进。

---

## 四、具体恢复策略

### 场景 1：子 agent 超时

处理顺序：
1. 判断是否值得一次重试
2. 若任务本身很短，墨影可直接接管收口
3. 若任务仍复杂，标记为 `failed` 或 `blocked`
4. 向用户说明：哪一步超时、是否已降级处理

建议：
- 同一子任务默认不超过 1 次重试
- 连续超时不应无限重派

---

### 场景 2：工具失败

处理顺序：
1. 记录失败工具与失败位置
2. 判断是否存在替代路径
3. 若可替代，则换路径继续
4. 若不可替代，则进入 `blocked` 或 `failed`

要求：
- 不把工具报错原样堆给用户
- 要给出“已尝试 / 未尝试 / 建议下一步”

---

### 场景 3：依赖缺失

典型情况：
- 没权限
- 文件不存在
- 环境没装好
- 账号未登录
- 配置缺失

处理顺序：
1. 明确缺的是哪种依赖
2. 判断是否可由墨影补齐
3. 若不可补齐，进入 `blocked`
4. 对用户输出最小补充动作

---

### 场景 4：结果冲突

处理顺序：
1. 不把冲突原样抛给用户
2. 回看证据、验证范围与风险等级
3. 必要时让问隙做第二轮验证
4. 最终由墨影裁决

原则：
- 冲突是主 Agent 的责任，不是用户的负担

---

### 场景 5：部分成功

处理顺序：
1. 拆清已完成项
2. 拆清失败项
3. 判断是否能先交付部分结果
4. 若能交付，则明确“已完成 / 未完成 / 风险”
5. 若不能交付，则进入 `failed` 或 `blocked`

---

### 场景 6：风险升级

处理顺序：
1. 立即停止自动推进
2. 把状态切到 `review` 或 `stopped`
3. 由墨影重新审议
4. 必要时向用户说明为什么要暂停

典型情况：
- 原本以为是普通修改，做着做着涉及部署、权限、密钥、删除覆盖

---

### 场景 7：交接无效

典型情况：
- “做好了”
- “你看下”
- 没有产物位置
- 没有验证方式
- 没有风险说明

处理顺序：
1. 不进入 Released
2. 退回为 `handoff` 或 `review`
3. 补齐交接卡后再继续

---

## 五、建议状态切换

### 超时
`in_progress -> failed`
或
`in_progress -> review`（若墨影接手）

### 工具失败
`in_progress -> blocked`
或
`in_progress -> failed`

### 依赖缺失
`in_progress -> blocked`

### 结果冲突
`handoff -> review`

### 部分成功
`in_progress -> handoff`
或
`in_progress -> review`

### 风险升级
`in_progress -> review`
或
`in_progress -> stopped`

---

## 六、对用户的最小输出要求

失败时，至少应告诉用户：
1. 哪一步失败
2. 原因是什么
3. 已完成什么
4. 目前状态是什么（failed / blocked / stopped）
5. 建议下一步是什么

---

## 七、一句话原则

> 多 Agent 系统的稳定，不取决于永不失败，而取决于失败后能否清楚降级、恢复和收口。
