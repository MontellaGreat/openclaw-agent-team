# FAQ

## 1. 这个项目是“必须多 agent”吗？
不是。

最小可用方式是：
- 主 Agent 先判断任务复杂度
- 简单任务直接执行
- 复杂任务再拆给附属 agent

如果目标 OpenClaw 没有真实子 agent，也可以先用主 Agent 内部分工模拟。

---

## 2. 主 Agent 为什么不直接当纯调度员？
因为纯调度会让简单任务也被过度拆解，增加成本和交互延迟。

本项目采用：
- 简单任务直接做
- 复杂任务再调度

这是效率和控制力更平衡的方案。

---

## 3. 为什么又引入了承枢？
因为复杂任务里真正消耗主 Agent 的，往往不是最终拍板，而是：
- 阶段推进
- handoff 回收
- 阻塞暴露
- 过程协调

承枢承担的是**编排协调劳动**，不是最终决策权。

---

## 4. 什么时候该拆任务？
当任务满足以下任一情况时，建议拆：
- 需要多个专项能力协作
- 需要调研后再写作/实现
- 需要代码实现再测试验证
- 需要多媒体调度
- 需要 structured ops 与人工判断协同
- 存在明显高风险或重大不确定性

---

## 5. 什么时候不该拆任务？
- 任务很短
- 主 Agent 自己能稳定完成
- 分发成本高于执行收益
- 用户只需要一个快速结论

---

## 6. 高风险任务包括哪些？
包括但不限于：
- 配置变更
- 权限变更
- 密钥处理
- 系统服务修改
- 删除覆盖
- 部署发布
- 正式对外发送
- 涉及隐私/金钱/法律事项

---

## 7. 为什么要引入任务生命周期？
因为多 agent 最怕“现在做到哪一步不清楚”。

生命周期让你知道：
- 任务刚进入
- 正在判断
- 已拆解
- 正在执行
- 正在交接
- 正在复核
- 已放行 / 已失败 / 已阻塞 / 已中止

---

## 8. handoff 为什么这么重要？
因为多 agent 最容易死在交接上。

一句“做好了你看下”不算交接。
真正的 handoff 至少要说清：
- 做了什么
- 结果是什么
- 怎么验证
- 有什么风险
- 下一步建议

---

## 9. 观象现在为什么不是视觉理解 agent 了？
因为本项目当前的结构缺口不在“看图”，而在：
- 日常运营
- 例行执行
- structured ops
- cron 驱动任务
- 稳定巡检与汇总

所以观象被重定义为：
- 观运行之象
- 做稳定、重复、结构化的执行任务

---

## 10. 问隙为什么被升级成 QA / 审查位？
因为“测试排障”如果只停留在复现和日志分析，链路还是不完整。

现在问隙还负责：
- QA 检查
- 风险补洞
- 放行建议

这样它才能成为实现链路中的质量门禁位。

---

## 11. 片场为什么是调度中枢，不直接当万能多媒体 agent？
因为多媒体任务类型差异很大：
- 生图
- 视频
- 修图

把它们混成一个万能角色，会导致：
- 输入要求不清
- 规则打架
- 调度混乱

所以片场负责判断和分发，子 agent 负责专项执行。

---

## 12. 并行派遣是不是越多越好？
不是。

只有在：
- 子任务彼此独立
- 能明显提速
- 汇总成本低于收益
- 不会放大风险

时才值得并行。

---

## 13. 这个项目能和 control center 一起用吗？
可以。

推荐理解为：
- 本项目 = 制度层
- control center = 观察层

一个管“怎么协作”，一个管“怎么看运行”。

---

## 14. 如果我的 agent 名称不一样怎么办？
不需要硬凑同名。

你只需要做能力映射：
- 研究能力 -> 探针
- 内容成稿 -> 笔官
- 工程执行 -> 铁手
- QA / 审查 -> 问隙
- structured ops -> 观象
- 多媒体调度 -> 片场
- 编排协调 -> 承枢

重点是角色和边界，不是名字本身。
