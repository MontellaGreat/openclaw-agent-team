# FAQ

## 1. 这个项目是“必须多 agent”吗？
不是。

最小可用方式是：
- 主 Agent 先判断任务复杂度
- 简单任务直接执行
- 复杂任务再拆给附属 agent

如果目标 OpenClaw 没有真实子 agent，也可以先用主 Agent 内部分工模拟。

---

## 2. 主 Agent 为什么不直接当纯调度员？
因为纯调度会让简单任务也被过度拆解，增加成本和交互延迟。

本项目采用：
- 简单任务直接做
- 复杂任务再调度

这是效率和控制力更平衡的方案。

---

## 3. 什么时候该拆任务？
当任务满足以下任一情况时，建议拆：
- 需要多个专项能力协作
- 需要调研后再写作/实现
- 需要代码实现再测试验证
- 需要多媒体调度
- 存在明显高风险或重大不确定性

---

## 4. 什么时候不该拆任务？
- 任务很短
- 主 Agent 自己能稳定完成
- 分发成本高于执行收益
- 用户只需要一个快速结论

---

## 5. 高风险任务包括哪些？
包括但不限于：
- 配置变更
- 权限变更
- 密钥处理
- 系统服务修改
- 删除覆盖
- 部署发布
- 正式对外发送
- 涉及隐私/金钱/法律事项

---

## 6. 为什么要引入任务生命周期？
因为多 agent 最怕“现在做到哪一步不清楚”。

生命周期让你知道：
- 任务刚进入
- 正在判断
- 已拆解
- 正在执行
- 正在交接
- 正在复核
- 已放行 / 已失败 / 已阻塞 / 已中止

---

## 7. handoff 为什么这么重要？
因为多 agent 最容易死在交接上。

一句“做好了你看下”不算交接。
真正的 handoff 至少要说清：
- 做了什么
- 结果是什么
- 怎么验证
- 有什么风险
- 下一步建议

---

## 8. 片场为什么是调度中枢，不直接当万能多媒体 agent？
因为多媒体任务类型差异很大：
- 生图
- 视频
- 修图

把它们混成一个万能角色，会导致：
- 输入要求不清
- 规则打架
- 调度混乱

所以片场负责判断和分发，子 agent 负责专项执行。

---

## 9. 视频换人为什么不单独做第 4 个子 agent？
当前项目里，视频换人被并入了 `片场-视频`。

原因：
- 同属视频能力域
- 规则与视频生成体系高度相关
- 独立成第 4 个 agent 会增加复杂度而收益有限

---

## 10. 并行派遣是不是越多越好？
不是。

只有在：
- 子任务彼此独立
- 能明显提速
- 汇总成本低于收益
- 不会放大风险

时才值得并行。

---

## 11. 这个项目能和 control center 一起用吗？
可以。

推荐理解为：
- 本项目 = 制度层
- control center = 观察层

一个管“怎么协作”，一个管“怎么看运行”。

---

## 12. 如果我的 agent 名称不一样怎么办？
不需要硬凑同名。

你只需要做能力映射：
- 调研能力 -> 探针
- 写作能力 -> 笔官
- 视觉理解 -> 观象
- 工程执行 -> 铁手
- 测试排障 -> 问隙
- 多媒体调度 -> 片场

重点是角色和边界，不是名字本身。
