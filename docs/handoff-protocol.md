# Handoff Protocol

本文件定义多 Agent 之间的标准交接协议（handoff contract）。

---

## 一、为什么需要 handoff

多 Agent 最容易烂的地方，不是不会做，而是：
- 做完了但说不清
- 产物交出来了但找不到
- 下一个 agent 不知道怎么接
- 墨影拿到一堆碎片，无法快速裁决

所以必须有统一交接协议。

---

## 二、标准交接内容

任意附属 agent 向墨影、承枢或下游 agent 交接时，至少包含：

1. **做了什么**
2. **产物是什么 / 在哪里**
3. **怎么验证**
4. **已知问题 / 风险**
5. **下一步建议**

---

## 三、推荐交接模板

```text
1. 任务边界
2. 已完成内容
3. 产物/结果位置
4. 验证方式
5. 已知问题/风险
6. 未完成项（如有）
7. 下一步建议
```

---

## 四、按角色的交接重点

### 承枢
重点写清：
- 当前阶段
- 已回收的 handoff
- 当前阻塞 / 依赖
- 需要墨影裁决的点

### 笔官
重点写清：
- 成稿版本
- 是否还有事实不确定点
- 是否可直接对外使用

### 探针
重点写清：
- 结论可信度
- 推荐路径
- 证据不足处

### 观象
重点写清：
- 执行的是哪类 structured ops 任务
- 产出位置
- 是否发现异常
- 是否需要升级给承枢或墨影

### 铁手
重点写清：
- 改了什么
- 没改什么
- 如何验证
- 有无回滚点

### 问隙
重点写清：
- 验证范围
- 未覆盖范围
- 风险等级
- QA 结论
- 是否建议放行

### 片场
重点写清：
- 任务类型判断
- 用了哪个子 agent
- 当前是草案还是正式产物
- 风险与限制

---

## 五、坏交接示例

### 坏例子
- “做好了，你看下。”
- “代码在里面。”
- “大概没问题。”

### 好例子
- “已完成登录页样式修复。改动集中在 `src/login.css` 与 `src/login.tsx`。建议运行 `npm test` 和手动检查三种屏宽。已知问题：深色模式未覆盖。下一步：问隙做边界验证与 QA 检查。”

## 七、一句话原则

> 交接不是一句“做完了”，而是让下一个人能无缝接上。
