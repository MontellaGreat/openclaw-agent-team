# Release Discipline

本文件定义多 Agent 体系中“什么时候只是做完”“什么时候可进入审议”“什么时候才算可以放行”。

目标：
- 把 `draft / reviewable / releasable / released` 区分清楚
- 防止把“有产出”误当成“可发布 / 可执行 / 可放行”
- 让墨影的最终放行判断有稳定门槛，而不是临场拍脑袋

---

## 一、为什么需要 release discipline

前两层已经解决了：
- 谁做什么
- 谁不该做什么
- 怎么委派
- 怎么交接
- 怎么保持 session 干净

但如果没有 release discipline，还是会出现：
- 有草案就被当成正式版
- 有实现就被当成可上线
- 有审查意见就被当成最终结论
- review 做了，但“能不能放”仍然口径不一

所以第三层必须补：
**放行纪律。**

---

## 二、四种结果状态

### 1. `draft`
定义：
- 已经有初步产出
- 但仍明显处于草案、方案、方向稿、执行稿阶段

典型例子：
- 笔官的初稿
- 铁手的改动方案
- 片场的方向稿
- 高风险动作的执行稿

特点：
- 可讨论
- 可继续加工
- 不应默认发布 / 执行 / 对外使用

---

### 2. `reviewable`
定义：
- 产出已经足够进入 review
- 具备最小审查条件
- 但尚未被认定为可放行

进入 `reviewable` 的最小条件：
- 目标明确
- 产出可检查
- 风险已说明
- 未验证项已标注
- 有基本 handoff

特点：
- 可以交给问隙或墨影做审查
- 不等于已通过

---

### 3. `releasable`
定义：
- 已经通过必要 review
- 已满足当前场景的最小放行条件
- 可以提交给墨影做最终 release decision

注意：
- `releasable` 不是 `released`
- 它表示“具备放行资格”，不表示“已经放行”

---

### 4. `released`
定义：
- 已由墨影作出最终放行判断
- 当前结果被视为正式交付 / 正式对外 / 正式执行

注意：
- 只有墨影能让结果进入 `released`
- specialist 和承枢都不能越权直接宣布 `released`

---

## 三、最小放行链路

推荐把结果流转理解成：

`draft -> reviewable -> releasable -> released`

并不是每个任务都必须显式写四个状态，
但在高风险任务、正式外发任务、实现与 QA 链路里，最好明确区分。

---

## 四、什么结果只能停在 draft

以下情况，默认只能停在 `draft`：
- 高风险配置修改的执行稿
- 正式对外内容的未审稿
- 正式素材的方向稿
- 没有验证方式的实现结果
- 风险说明不完整的产出
- handoff 明显缺项的结果

一句话：
**只要还明显需要补验证、补风险说明、补主控判断，就不要把它抬到 `reviewable` 以上。**

---

## 五、进入 reviewable 的门槛

要进入 `reviewable`，至少满足：
1. 任务边界清楚
2. 当前产出可直接检查
3. 风险 / 不确定点已标注
4. 交接最小字段已具备
5. 若有产物，位置可追踪

若缺以上任意关键项：
- 退回补交接
- 不进入 review

---

## 六、进入 releasable 的门槛

要进入 `releasable`，至少满足：
1. 必要 review 已完成
2. 关键风险已说明
3. 未验证项已明确
4. 若需要验证，已完成或已明确豁免理由
5. 若是可执行动作，已有回滚或止损思路
6. 没有关键冲突未裁决

注意：
- `releasable` 可以理解成“可提交墨影拍板”
- 它不是“系统已经默认同意”

---

## 七、进入 released 的门槛

进入 `released`，必须同时满足：
1. 已达到 `releasable`
2. 墨影已做最终审议
3. 墨影确认当前结果符合目标与风险要求
4. 若属于高风险动作，墨影明确允许进入执行 / 发布

---

## 八、哪些任务必须显式走 release discipline

以下任务不应只靠口头判断，建议显式区分 `draft / reviewable / releasable / released`：
- 配置变更
- 权限变更
- 密钥相关动作
- 系统服务修改
- 删除覆盖
- 部署发布
- 正式对外内容
- 正式素材 / 正式视频
- 用户明确要求“能不能放行”

---

## 九、角色在放行链路中的位置

### 墨影
- 唯一 release decision owner
- 决定是否进入 `released`

### 承枢
- 协调链路
- 回收阶段结果
- 标记当前在哪个放行阶段
- 不能替代墨影做最终放行

### 问隙
- 提供 QA / 审查结论
- 可以建议 `not_releasable` / `releasable_with_risks`
- 不能直接宣布 `released`

### 铁手 / 笔官 / 探针 / 观象 / 片场
- 提供各自专业产出
- 可以把结果交到 `draft` 或 `reviewable`
- 不拥有最终放行权

---

## 十、建议的 release decision 结果值

建议墨影在最终放行时，用以下口径之一：
- `released`
- `released_with_risks`
- `not_released`
- `stopped_pending_review`

这样比模糊的“应该差不多可以”更可追踪。

---

## 十一、一句话原则

> 能审不等于能放；能放也必须由墨影来放。
