# Troubleshooting

## 1. 任务被过度拆解
### 表现
- 很小的任务也被派给多个 agent
- 响应慢
- 用户只问一句话却走了一整套流程

### 原因
- 主 Agent 没先判断简单/复杂
- 调度意识不足

### 解决
- 先按 `docs/task-complexity-levels.md` 分级
- L1 任务默认直接执行
- 只有 L3/L4 才优先考虑拆解

---

## 2. 主 Agent 像中转站，不像总控
### 表现
- 什么结果都原样转发
- 不做裁决
- 不做最终审议

### 原因
- 墨影没有执行“回收结果后裁决”职责

### 解决
- 按 `docs/main-agent-decision-flow.md` 执行
- 汇总后检查：目标是否满足、是否冲突、是否有未验证项、是否可放行

---

## 3. 多 agent 结果冲突
### 表现
- 探针说可行，问隙说风险高
- 铁手说做完了，问隙说不建议放行

### 解决
- 不要把冲突原样抛给用户
- 由墨影按以下优先级裁决：
  1. 已验证证据优先
  2. 实测优先
  3. 低风险可回滚优先
  4. 高风险任务中，问隙的风险判断权重更高

---

## 4. 交接质量很差
### 表现
- “做好了你看下”
- 不知道改了什么
- 不知道怎么验证

### 解决
- 强制使用 `docs/handoff-protocol.md`
- 交接至少写清：做了什么、结果是什么、怎么验证、有什么风险、下一步建议

---

## 5. 片场乱接任务
### 表现
- 文案没定稿就开始做正式图
- 修图任务被当成生图
- 视频换人被当成普通文生视频

### 解决
- 片场先判断任务类型
- 再套子规则文件：
  - `rules/pianchang-image-input.md`
  - `rules/pianchang-video-input.md`
  - `rules/pianchang-edit-input.md`

---

## 6. 高风险任务被直接落地
### 表现
- 改配置直接执行
- 正式对外内容未经审议就发出

### 解决
- 立即恢复高风险规则
- 相关任务只能先出方案/草案/执行稿
- 必须进入 Review，再由墨影放行

---

## 7. 并行派遣反而更乱
### 表现
- 大家都在做，但最后更慢
- 任务之间相互依赖却被硬拆

### 解决
- 检查是否真的满足并行条件
- 如果存在明显前后依赖，改回串行
- 并行不是默认动作，只是可选策略

---

## 8. 没有真实子 agent 怎么办？
### 解决
- 先用 prompt-only 模式启动
- 由主 Agent 内部分工模拟
- 等环境允许时，再逐步接入真实子 agent

---

## 9. 项目迁移到别的 OpenClaw 后不顺手
### 原因
- agent 名称不同
- 路径不同
- prompt 注入位置不同

### 解决
- 使用 `INSTALL.md`
- 做角色能力映射，而不是执着于同名
- 先跑一个简单任务 + 一个复杂任务验证

---

## 10. 任务卡住了
### 表现
- 长时间没有结果
- 不知道卡在哪个环节

### 解决
- 回看生命周期：任务当前是 `Assigned`、`In Progress`、`Handoff` 还是 `Review`
- 检查是否缺输入、缺权限、缺依赖
- 必要时将状态标记为 `Blocked`
