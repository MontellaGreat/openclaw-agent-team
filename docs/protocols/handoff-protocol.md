# Handoff Protocol

本文件定义多 Agent 协作中真正可执行的交接协议。

目标不是写“做完了”，而是让：
- 主 Agent 能快速吸收结果
- 下一个接手的 agent 能无缝继续
- 承枢能稳定回收阶段产物
- 高风险链路不会因为交接含糊而误放

---

## 一、handoff 的作用

handoff 不是礼貌汇报，而是运行协议的一部分。

一个有效 handoff 至少要解决 4 件事：
1. 说清任务边界
2. 说清已经完成了什么
3. 说清结果是否可信、如何验证
4. 说清接下来该由谁接、是否需要主 Agent 裁决

如果做不到这 4 件事，handoff 就是不合格的。

---

## 二、最小 handoff 必填字段

任意附属 agent 向墨影、承枢或下游 agent 交接时，至少要带：

1. 任务边界
2. 已完成内容
3. 核心结果 / 产出
4. 产物位置（如适用）
5. 验证方式
6. 风险 / 不确定点
7. 未完成项（如有）
8. 是否需要主 Agent 决策
9. 下一步建议

如果缺少其中关键字段，默认视为：
- 交接不完整
- 不可直接 Released
- 需要补充或退回 Review / Handoff

---

## 三、标准 handoff 模板

```text
任务边界：
已完成内容：
核心结果 / 产出：
产物位置：
验证方式：
风险 / 不确定点：
未完成项：
是否需要主 Agent 决策：
下一步建议：
```

要求：
- 尽量具体，不写空话
- 不要只写“已完成”
- 不要把猜测写成确认结论
- 不要省略验证与风险

---

## 四、主 Agent 回收 handoff 时的检查动作

墨影收到 handoff 后，不直接转给用户。

默认先检查：
1. 任务边界是否清楚
2. 输出是否真正回答了原任务
3. 是否缺验证方式
4. 是否存在高风险项
5. 是否有明显不确定点未标出
6. 是否需要补派单
7. 是否需要进入 review

若 handoff 不完整：
- 退回补齐
- 或由承枢继续回收整理
- 不直接当成完成

---

## 五、按角色的 handoff 重点

## 5.1 承枢
重点写清：
- 当前推进到哪个阶段
- 已回收哪些 handoff
- 当前阻塞 / 依赖
- 需要墨影裁决什么
- 下一阶段建议怎么推进

## 5.2 探针
重点写清：
- 研究结论
- 证据强弱
- 推荐路径
- 仍待验证点
- 哪些结论不能直接当成定论

## 5.3 笔官
重点写清：
- 成稿版本是什么
- 面向谁
- 是否可直接使用
- 哪些事实仍需核对
- 是否只是草案

## 5.4 铁手
重点写清：
- 改了什么
- 没改什么
- 影响范围
- 如何验证
- 回滚点或保守替代方案

## 5.5 问隙
重点写清：
- 验证范围
- 未覆盖范围
- 发现的问题
- 风险等级
- 是否建议放行

## 5.6 观象
重点写清：
- 执行了哪类 structured ops 任务
- 产出位置
- 是否有异常
- 是否需要升级为人工判断

## 5.7 片场
重点写清：
- 当前任务类型判断
- 使用了哪个子能力
- 当前产物是草案还是正式版本
- 素材限制与风险
- 下一轮迭代建议

---

## 六、坏交接与好交接

### 坏交接
- “做好了，你看下。”
- “代码在里面。”
- “大概没问题。”
- “我觉得应该可以发。”

### 好交接
- “已完成登录页样式修复。改动集中在 `src/login.css` 和 `src/login.tsx`。已验证桌面端正常，移动端未完整验证。当前风险是深色模式未覆盖。建议下一步由问隙做边界验证，再由墨影决定是否放行。”

---

## 七、handoff 与 released 的关系

必须明确：
- handoff 不等于完成
- 有产物不等于可放行
- 交接完成只是进入主 Agent 汇总、裁决、审议的前置步骤

特别是以下情况，不能因 handoff 存在就默认 Released：
- 高风险任务
- 未验证任务
- 结果冲突任务
- 仍有明显缺口的任务

---

## 八、一句话原则

> handoff 的目标不是“告诉别人我做过了”，而是让主 Agent 或下一个执行单元能立刻接住，并且知道结果是否可信、风险在哪里、下一步该怎么走。
