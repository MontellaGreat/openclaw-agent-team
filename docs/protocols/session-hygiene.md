# Session Hygiene

本文件定义多 Agent 协作中的 session 使用纪律。

目标：
- 避免多 agent 会话串味
- 避免一个 session 里混进多个无关目标
- 避免历史污染当前判断
- 让委派、回收、追问、归档更可控

---

## 一、为什么需要 session hygiene

真实多 agent 协作一旦进入运行时，最容易出现的问题不是不会 spawn，而是：
- 旧上下文污染新任务
- 子任务边界变形
- 一个 specialist 会话里塞进多个目标
- 结果回收时分不清哪段输出属于哪个任务

所以 session 需要纪律，而不是“能开就开、能复用就复用”。

---

## 二、总原则

### 原则 1：一个 session 最好只承载一个稳定目标
如果目标变了、边界变了、角色变了，就不该默认沿用旧 session。

### 原则 2：能短闭环就短闭环
可快速完成的子任务，不要开成长线脏会话。

### 原则 3：复用 session 必须有收益
复用的前提不是省一次 spawn，而是：
- 连续任务同域
- 共享上下文明显有价值
- 不会引入混淆

### 原则 4：状态比数量更重要
不是 session 越少越好，而是每个 session 的职责要清楚。

---

## 三、什么时候应该新开 session

建议新开 session 的情况：
- 新任务目标和旧任务无直接连续性
- 角色切换明显（如从研究切到实现）
- 需要隔离风险
- 需要并行独立执行
- 当前 session 已被大量历史污染
- 任务已进入新阶段，且阶段间边界清晰

典型例子：
- 探针完成调研后，不把同一 session 直接改造成铁手实现位
- 问隙做 QA 时，不直接复用铁手的实现会话当验证会话

---

## 四、什么时候可以复用 session

可考虑复用的情况：
- 同一角色
- 同一目标
- 同一阶段或紧邻阶段
- 上下文连续且不会造成歧义

典型例子：
- 探针连续补两轮同一主题研究
- 铁手在同一缺陷修复链路里继续补一个小 patch
- 观象持续执行同一类 structured ops 例行任务

---

## 五、什么时候必须停止复用

出现以下情况时，应停止复用旧 session：
- scope 漂移明显
- 任务目标改变
- 历史输出已经造成理解偏差
- specialist 被要求做超出原角色的工作
- 会话里堆积了太多互不相干的回合

一句话判断：
**如果需要先花很多话解释“现在和刚才不是一回事”，那通常就该新开。**

---

## 六、承枢与 session 管理

承枢在多步骤任务中，应承担最小 session hygiene 职责：
- 记录当前有哪些活跃 session
- 标记每个 session 对应的角色与目标
- 判断某个子任务该复用还是新开
- 在回收 handoff 时注明来源 session
- 防止一个 specialist session 无限制漂移

承枢不需要管理所有底层细节，但必须对“哪个任务在哪个 session 里”有清晰感知。

---

## 七、结果回收时必须带上的信息

每次回收子任务结果时，建议至少带上：
- `task_id`
- `agent`
- `session_id` 或等价引用
- 当前状态
- 本轮结果摘要
- 下一步建议

没有来源标识的结果，后续很难做可靠复盘。

---

## 八、清理与归档建议

### 低价值会话
以下会话可尽快收口：
- 一次性问答型子任务
- 已明确失败且无继续价值的尝试
- 已被新 session 替代的旧会话

### 高价值会话
以下会话值得保留引用：
- 关键调研链路
- 关键实现链路
- 关键 QA / 风险审查链路
- 后续还会反复复用的 structured ops 会话

保留的不是“所有内容”，而是：
- 任务边界
- 关键结论
- 产物位置
- 风险与未完成项

---

## 九、和 A2A 协议的关系

A2A delegation protocol 解决：
- 怎么派
- 怎么回
- 怎么标状态

session hygiene 解决：
- 这个任务该在哪个会话里跑
- 什么时候该新开
- 什么时候该停用旧会话

两者缺一不可。

---

## 十、一句话原则

> session 不是越省越好，而是越干净越好。
