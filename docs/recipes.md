# recipes.md

这份文件不是讲制度，而是给你几种高频任务的直接用法。

---

## 1. 调研 + 成稿

### 用户任务
- “调研一下 A 和 B 哪个更适合我们，再写一版汇报给我。”

### 推荐流转
- 探针：做调研、方案比较、风险点整理
- 笔官：把调研结论整理成可直接使用的汇报稿
- 墨影：统一裁决、补充结论、决定最终交付

### 主 Agent 预期动作
- 先判断为 L3
- 先说明准备怎么拆
- 回收探针与笔官结果后统一汇总

---

## 2. structured ops / 巡检任务

### 用户任务
- “每天早上检查一次运行状态，发现异常就告诉我。”

### 推荐流转
- 观象：执行 structured ops、巡检、汇总
- 若出现异常：承枢协调后续链路
- 必要时：铁手补开发，问隙补 QA / 风险判断
- 墨影：做最终决策

### 主 Agent 预期动作
- 先判断是否属于 L2 或 L3
- 低风险稳定任务交给观象
- 异常时再升级为多角色协作

---

## 3. 代码修改 + 放行前检查

### 用户任务
- “修这个 bug，然后告诉我能不能放行。”

### 推荐流转
- 铁手：改代码 / 脚本 / 配置
- 问隙：复现、验证、边界检查、QA 审查、回归建议
- 墨影：决定是否建议放行

### 主 Agent 预期动作
- 判断为 L3
- 不把“已修改”直接等同于“可放行”
- 最终结论由墨影给出

---

## 4. 配图请求

### 用户任务
- “根据这份公众号内容，给我一版配图思路。”

### 推荐流转
- 若只是思路：墨影或笔官可直接给出
- 若要进入多媒体执行：片场先判断是生图、修图还是视频

### 主 Agent 预期动作
- 文案未定稿时，不盲目推进生成
- 风格未明确时，先补约束

---

## 5. 高风险配置修改

### 用户任务
- “把线上服务配置改掉并重启。”

### 推荐流转
- 铁手：给执行稿或改动方案
- 问隙：补充检查项、QA 与回滚建议
- 墨影：决定是否放行

### 主 Agent 预期动作
- 直接判定为 L4
- 不默认直接执行到底
- 先给方案 / 执行稿 / 检查清单

---

## 6. 正式对外发布

### 用户任务
- “把这份公告直接发出去。”

### 推荐流转
- 笔官：整理正式文案
- 必要时探针：核实事实
- 问隙：补审查（如适用）
- 墨影：最终审议是否可发

### 主 Agent 预期动作
- 直接判定为高风险
- 不让附属 Agent 越权对外承诺
- 明确“当前是草案 / 当前建议可发 / 当前不建议发”

---

## 一句话原则

高频任务不要靠临场发挥。
先判断属于哪类模式，再套用合适的协作路径。
