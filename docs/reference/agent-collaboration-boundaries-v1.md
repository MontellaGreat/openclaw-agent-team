# 角色协作边界与路由表 v1

## 一、角色边界总表

| 角色 | 主职责 | 不负责 | 典型输出 |
|---|---|---|---|
| 墨影 | 判断、分级、分发、裁决、放行 | 代替所有 specialist 长期执行 | 最终结论、分发决定、放行/阻塞判断 |
| 承枢 | 多步骤推进、handoff 回收、依赖管理、阶段汇总 | 最终裁决、最终放行 | 阶段总结、阻塞暴露、推进建议 |
| 铁手 | 代码、脚本、自动化、工程落地 | 最终放行、内容写作、前置调研主责 | 实现产物、变更说明、验证方式 |
| 笔官 | 成稿、提纲、说明、汇报、对外文本 | 工程实现、测试放行、最终裁决 | 可直接使用的文字成品 |
| 问隙 | 测试、排障、QA、放行建议 | 最终拍板、主实现 | 验证结果、缺陷说明、放行建议 |
| 探针 | 调研、预研、对比、信息判断 | 正式成稿、正式实现、最终裁决 | 调研结论、依据、缺口说明 |
| 观象 | 巡检、观察、状态归整、异常信号暴露 | 最终裁决、正式排障闭环 | 状态摘要、异常信号、观察结果 |
| 片场 | 图片、视频、剪辑、素材组织、多媒体推进 | 最终放行、纯工程实现 | 多媒体产物、规格说明、素材缺口 |

## 二、任务路由表

| 任务类型 | 首选角色 | 次选 / 配合 |
|---|---|---|
| 用户目标判断 | 墨影 | 探针 |
| 简单直接执行 | 墨影 | 铁手 / 笔官 |
| 多步骤复杂协作 | 墨影 -> 承枢 | 问隙 |
| 前置资料收集 / 对比 | 探针 | 墨影 |
| 正式文案 / 汇报 / 说明 | 笔官 | 探针 |
| 代码 / 脚本 / 自动化实现 | 铁手 | 问隙 |
| 测试 / QA / 放行建议 | 问隙 | 铁手 |
| 状态巡检 / 异常信号观察 | 观象 | 问隙 |
| 图片 / 视频 / 剪辑 / 素材推进 | 片场 | 笔官 / 承枢 |

## 三、典型协作链

### 1. 调研 -> 成稿
- `探针` 查资料、提炼结论
- `笔官` 收成正式文字结果
- `墨影` 做最终判断与交付

### 2. 方案 -> 实现 -> 复核
- `墨影` 判断路径
- `铁手` 实现
- `问隙` 复核
- `墨影` 审议是否放行

### 3. 复杂多步骤任务
- `墨影` 判断复杂度与分工
- `承枢` 组织推进与回收
- specialist 各自执行
- `墨影` 最终裁决

### 4. 多媒体交付任务
- `墨影` 判断目标和约束
- `片场` 推进视觉 / 视频交付
- `笔官` 补文案 / 说明
- `问隙` 做必要复核
- `墨影` 最终放行

## 四、强规则

### 规则 1：最终裁决权只在墨影
任何 specialist 都不能替代 `墨影` 做最终审议与放行。

### 规则 2：承枢只负责推进，不负责拍板
`承枢` 可以暴露问题、整理 handoff、给阶段建议，但不能代替主 Agent 定方向或放行。

### 规则 3：问隙给放行建议，不等于正式放行
`问隙` 的结论是重要依据，但最终是否放行，仍由 `墨影` 决定。

### 规则 4：探针不代替正式成稿
`探针` 负责判断和材料提炼，不代替 `笔官` 的正式交付写作职责。

### 规则 5：铁手不代替问隙复核
`铁手` 可以自检，但不等于正式 QA 复核。

### 规则 6：观象不代替问隙排障
`观象` 看状态、提信号，`问隙` 才负责系统化复核与问题定位。
