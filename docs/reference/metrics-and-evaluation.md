# Metrics and Evaluation

本文件定义多 Agent 团队方案的最小评估指标。

目标：
- 判断这套团队设计是否真的提升了效率、质量和风险控制
- 避免只看“是否分发了任务”，不看“是否更好地解决了任务”
- 为后续控制中心、复盘和持续优化提供统一口径

---

## 一、为什么需要评估

如果没有评估，多 Agent 方案容易陷入以下错觉：
- 任务拆了很多，看起来很忙，但整体更慢
- 调度很热闹，但质量并没有提升
- 交接很多，但返工更多
- 高风险任务看似都经过 review，实际仍然放行了不稳结果

所以必须回答四个问题：
1. 拆分是否真的带来收益
2. 交接是否真的有效
3. review 是否真的拦住了问题
4. 主 Agent 的调度是否越来越准确

---

## 二、指标分层

建议把指标分成四层：
- 调度层
- 交付层
- 风险层
- 运行层

---

## 三、调度层指标

### 1. 直做命中率
定义：
- 原本不值得拆的任务，是否被墨影正确判定为直接执行

意义：
- 衡量墨影是否避免了“为了多 agent 而多 agent”

### 2. 拆分收益率
定义：
- 被拆分的任务中，有多少真的带来了：
  - 更快
  - 更稳
  - 更专业
  - 更低风险

意义：
- 衡量分发是否有真实收益

### 3. 并行有效率
定义：
- 被并行派发的任务中，有多少确实缩短了总耗时且没放大风险

意义：
- 衡量并行规则是否用对了地方

---

## 四、交付层指标

### 1. 一次通过率
定义：
- 附属 agent 交付后，无需返工即可进入 Released 的比例

意义：
- 衡量角色职责与 I/O 契约是否清楚

### 2. handoff 返工率
定义：
- 因交接不完整、格式不清、产物不明而被退回的比例

意义：
- 衡量 handoff 协议是否真正落地

### 3. 产物可用率
定义：
- 交付产物是否能被下游 agent 或墨影直接使用

意义：
- 衡量协作链条是否顺畅

---

## 五、风险层指标

### 1. review 拦截率
定义：
- 进入 Review 的任务中，有多少被识别出问题、风险或未验证项

意义：
- 衡量 quality gate 是否真的在起作用

### 2. 高风险复核覆盖率
定义：
- 高风险任务中，有多少确实经过墨影复核

意义：
- 衡量“高风险必审”是否落实

### 3. 错放率
定义：
- 已 Released 的任务里，后续被发现本不该放行的比例

意义：
- 衡量最终放行质量

---

## 六、运行层指标

### 1. 子任务超时率
定义：
- 被真实派发的子任务中，超时的比例

意义：
- 衡量 runtime 稳定性与派发质量

### 2. 阻塞率
定义：
- 任务进入 `blocked` 的比例

意义：
- 衡量依赖管理与前置判断能力

### 3. 恢复成功率
定义：
- 任务进入失败/阻塞路径后，通过换路、重试或降级仍然成功收口的比例

意义：
- 衡量失败恢复协议是否有效

---

## 七、建议复盘频率

### 每日轻复盘
关注：
- 是否有明显超时
- 是否有明显错路由
- 是否有高价值失败样本

### 每周复盘
关注：
- 哪类任务值得拆
- 哪类任务不值得拆
- 哪个 agent 常见交接问题最多
- 哪类 review 最容易拦下问题

### 每月复盘
关注：
- 团队规则是否需要调整
- 是否要新增/删除某类角色职责
- 是否需要升级控制台或自动化支持

---

## 八、一句话原则

> 多 Agent 团队不是看“拆得多不多”，而是看“是否更快、更稳、更可控”。
