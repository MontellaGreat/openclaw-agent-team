# Roster Discipline

本文件定义多 Agent 团队的编组纪律。

目标：
- 解决“角色都在，但谁都能上”的混乱
- 明确哪些角色常驻、哪些角色按需启用
- 明确哪些角色可以串联、哪些角色不宜合并
- 让团队结构从静态名单升级成可执行 roster

---

## 一、为什么需要 roster discipline

仅有角色定义还不够。

如果没有 roster discipline，常见问题是：
- 所有角色看起来都可用，但没有主次
- 同一个任务里拉了太多角色，汇总成本过高
- 某些本该分开的职责被同一个链路吞掉
- 主 Agent 因为缺编组规则，只能临场拍板

所以第三层要补：
**谁默认在场，谁按需出现，谁不该混成一坨。**

---

## 二、三类 roster

### 1. 常驻层（Core Roster）
默认长期存在：
- `墨影`
- `承枢`

说明：
- `墨影` 是唯一总控
- `承枢` 是复杂任务中的默认协调位
- 二者构成组织主轴

### 2. 专项层（Specialist Roster）
按任务类型启用：
- `探针`
- `笔官`
- `铁手`
- `问隙`
- `观象`
- `片场`

说明：
- 不是每个任务都该全员出场
- 专项层应该按目标拉起，而不是按热闹拉满

### 3. 片场子编组（Media Sub-Roster）
只在多媒体任务内启用：
- `片场-生图`
- `片场-视频`
- `片场-修图`

说明：
- 它们不应越过片场直接成为团队主角色
- 只在片场判断后按需挂载

---

## 三、默认编组规则

### 规则 1：L1 任务默认不拉专项层
- 由墨影直接完成
- 不为了“像团队”而开 roster

### 规则 2：L2 任务默认只拉一个 specialist
典型：
- 研究类 -> 探针
- 写作类 -> 笔官
- 工程类 -> 铁手
- QA / 审查类 -> 问隙
- structured ops -> 观象
- 多媒体类 -> 片场

### 规则 3：L3 任务默认是“墨影 + 承枢 + 必要 specialist”
- 墨影：总控与裁决
- 承枢：链路协调
- specialist：按阶段接入

### 规则 4：L4 任务默认必须有 review 位
高风险任务不应只有执行位。

建议最小链路：
- 工程高风险：`墨影 + 铁手 + 问隙`
- 内容高风险：`墨影 + 笔官 (+ 探针)`
- structured ops 异常升级：`墨影 + 观象 + 必要补位角色`

---

## 四、默认串联关系

### 稳定串联 1：探针 -> 笔官
适用：
- 调研后成稿
- 资料先行的汇报、方案、文章

### 稳定串联 2：铁手 -> 问隙
适用：
- 代码修改
- 自动化调整
- 配置执行稿

### 稳定串联 3：观象 -> 承枢 -> 墨影
适用：
- structured ops 发现异常
- 巡检结果进入人工决策链路

### 稳定串联 4：笔官 -> 片场
适用：
- 文案定稿后进入海报 / 视频 / 素材生产

---

## 五、默认互斥或不宜合并的职责

### 1. 执行位 vs QA 位
- `铁手` 与 `问隙` 可以在极小任务中由一人完成最小自测
- 但在正式放行链路里，不宜把“实现”和“QA 放行建议”长期合成一个角色

### 2. 编排位 vs 最终裁决位
- `承枢` 不应长期兼任墨影
- 否则会把协调判断和最终放行混在一起

### 3. structured ops 位 vs 研究判断位
- `观象` 不应逐步滑成探针
- 观象负责稳定执行，不负责路线判断

### 4. 片场调度位 vs 全部媒体执行位
- 片场可以小规模补位
- 但长期不应把生图 / 视频 / 修图全吞为一个混合角色

## 六、按需启用规则

### 探针
当任务涉及：
- 资料不明
- 方案对比
- 决策依据不足

再启用。

### 笔官
当任务涉及：
- 正式文稿
- 表达清理
- 结构化文本成稿

再启用。

### 铁手
当任务涉及：
- 代码 / 脚本 / 自动化 / 配置实现

再启用。

### 问隙
当任务涉及：
- QA / 放行建议
- 边界验证
- 高风险检查

优先启用。

### 观象
当任务涉及：
- 巡检
- cron
- 例行执行
- structured ops

再启用。

### 片场
当任务涉及：
- 生图
- 视频
- 修图
- 素材调度

再启用。

## 七、主 Agent 的 roster 判断问题单

墨影在决定编组前，至少先问：
1. 这个任务真的需要 specialist 吗？
2. 如果需要，是一个 specialist 还是多个？
3. 是否需要承枢协调，还是主会话内就能收口？
4. 是否存在必须分开的执行 / QA / 放行职责？
5. 当前拉进来的每个角色，是否真的带来收益？

只要有一个角色答不上“为什么必须在场”，就先别拉。

---

## 八、一句话原则

> roster 不是人越多越强，而是编组越准越稳。
