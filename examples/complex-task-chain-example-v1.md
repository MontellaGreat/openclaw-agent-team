# 复杂任务链路样例 v1

## 样例主题
把一个高风险配置变更，从用户请求一路推进到可审议状态。

## 阶段 1：主 Agent 判断
### 角色
- `墨影`

### 动作
- 判断这是配置类高风险任务
- 判断不能直接放行
- 分级为 `L4`
- 决定拆为：调研 / 实现 / 复核 / 审议

### 输出
- 给用户简洁说明为什么复杂
- 给 `铁手` 下实现任务
- 给 `问隙` 预留复核任务
- 视链路复杂度，让 `承枢` 接手推进

## 阶段 2：执行实现
### 角色
- `铁手`

### 动作
- 只在边界内完成配置修改
- 标出核心变更
- 说明验证方式和未验证项

### 输出
- 可 review 的变更结果
- 交给 `问隙` 做复核

## 阶段 3：测试复核
### 角色
- `问隙`

### 动作
- 检查格式、兼容性、关键路径
- 区分已验证与未验证项
- 给出放行建议

### 输出
- 有依据的复核结论
- 回主 Agent 或由 `承枢` 汇总

## 阶段 4：协调回收
### 角色
- `承枢`

### 动作
- 收齐 `铁手` 与 `问隙` 的 handoff
- 对齐阶段状态
- 标出阻塞与风险
- 暴露需要主 Agent 拍板的点

### 输出
- 阶段总结
- 是否进入最终审议条件判断

## 阶段 5：最终审议
### 角色
- `墨影`

### 动作
- 看证据是否足够
- 判断是否仍需补验证
- 决定放行 / 暂停 / 退回补充

### 输出
- 最终结论
- 风险说明
- 下一步动作

## 这个样例说明的核心规则
- 高风险任务不能由 specialist 直接放行
- `承枢` 负责推进和回收，不负责拍板
- `问隙` 负责放行建议，不负责最终放行
- `铁手` 负责实现，不代替 QA
- `墨影` 保留最终审议权
