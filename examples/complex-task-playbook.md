# Complex Task Playbook

本文件给出几套真实复杂任务的协作流程示例。

---

## 场景 1：调研 → 实现 → 验证/QA

### 用户目标
“帮我调研这个方案能不能做，然后把脚本改掉，再告诉我能不能放行。”

### 墨影处理
1. 判断为复杂任务
2. 需要技术预研 + 工程实现 + 验证/QA
3. 如链路较长，可交承枢推进编排
4. 先拆 3 段：
   - 探针：做预研和方案比较
   - 铁手：按推荐路径实现
   - 问隙：验证、QA 与放行建议
5. 汇总后由墨影给最终结论

---

## 场景 2：调研 → 写作 → 配图 → 终审

### 用户目标
“帮我写一篇公众号文章，并配一张图。”

### 墨影处理
1. 判断为复杂任务
2. 内容事实和成稿、配图都需要不同能力
3. 如链路较长，可由承枢负责回收 handoff
4. 拆分：
   - 探针：补事实与背景
   - 笔官：写正文
   - 片场：按定稿文案调度生图
5. 最后由墨影统一终审

---

## 场景 3：日常巡检 → 异常升级 → 决策

### 用户目标
“每天早上检查一次运行状态，发现异常就告诉我。”

### 墨影处理
1. 判断为 structured ops / 复杂任务
2. 观象承担例行执行与巡检
3. 若出现异常，交承枢协调后续链路
4. 需要开发修复则交铁手，需审查则交问隙
5. 最终由墨影决定是否升级处理

---

## 场景 4：文案 → 片场调度 → 多媒体产出

### 用户目标
“根据这个主题做一个海报，再给一个短视频方向。”

### 墨影处理
1. 判断为复杂任务
2. 先确认主题是否清楚，文案是否稳定
3. 若文案不稳，先让笔官补草案
4. 再交片场判断：
   - 海报 -> 片场-生图
   - 视频方向 -> 片场-视频
5. 片场汇总结果
6. 墨影决定是否继续迭代或放行

---

## 场景 5：高风险任务

### 用户目标
“帮我改配置并直接发布。”

### 墨影处理
1. 先判定为高风险任务
2. 不直接下放最终执行权
3. 可拆为：
   - 探针：评估改法和风险
   - 铁手：给执行稿或改动方案
   - 问隙：给 QA 与风险补充
4. 必要时由承枢推进链路
5. 墨影决定是否放行

---

## 场景 6：应该直接做、不拆解的任务

### 用户目标
“帮我把这句话润色一下。”

### 墨影处理
1. 判断为简单任务
2. 不拆
3. 直接执行

---

## 总结

复杂任务的核心不是“多叫几个人”，而是：
- 先判断是否真的复杂
- 复杂才拆
- 拆后边界清楚
- 必要时让承枢做编排协调
- 汇总后统一裁决
- 高风险必须复核
