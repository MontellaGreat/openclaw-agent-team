# taolun

## 讨论主题
围绕 `openclaw-agent-team` 项目本体，进行三轮多 Agent 讨论，目标是形成完整规划清单并最终沉淀为 `guihua.md`。

## 已知前提
- 当前进度已保存于 `jindu.md`
- 当前项目已具备：制度层、角色层、模板层、planning contract、runtime 对象层、SOP 层
- 当前仍待补：初始化层、验收层、planner 实现层、长期运行验证层

## 讨论规则
1. 每个 agent 在发言前先阅读 `jindu.md` 与 `taolun.md`
2. 每轮按顺序接力，后一个 agent 必须吸收前一个 agent 的观点
3. 每次只讨论“项目本体规划”，不讨论智能任务规划系统专项制作
4. 三轮讨论结束后，再单独开启 `taolun2.md`

### 探针
**对当前 v1 方向的判断**
- 当前方向大体成立：项目已从“角色设定仓”推进成“多 agent 协作模板仓”，下一步应收敛成一套可验证的最小运行产品。
- 但还差一个关键转身：现在“像系统”，但还没有用一条足够真实、足够窄的任务闭环证明自己“就是系统”。
- 因此下一步不是再补广度，而是做减法：先定义 v1 完成定义（DoD），再围绕一条最小真实任务场景倒推初始化、流转、验收、归档。

**给出的 v1 完成定义（DoD）**
- 可启动：新任务到来后，操作者能在不额外解释的前提下完成初始化，并明确启用角色、任务对象与流转路径。
- 可分发：主 Agent 能基于任务卡把任务交给承枢/执行位，且交接文本按模板产出、字段完整、责任清楚。
- 可收口：执行结束后，能形成明确的 handoff 与结果回收，不出现悬空状态。
- 可风控：低风险任务可轻量闭环；高风险任务能触发 review gate，并留下是否放行的依据。
- 可复盘：任务结束后，第三人只看 runtime 文件，就能看懂任务目标、分工过程、结果去向、是否完成。

**建议的最小真实任务场景**
- 不选“单纯写一篇新文档”，也不选“开发+测试+发布”的重型链路。
- 第一条实战闭环建议采用：`已有仓库中的单文件规则修订，并同步更新一处关联说明文档`。
- 理由：足够真实、风险适中、可清楚检验 main / chengshu / tieshou（必要时 wenxi）的角色边界。

- 把“最小真实任务场景”写成一页式闭环草案，明确必经节点、条件节点、结束条件

### 笔官
**结论**
- 我同意前面的收敛方向：项目本体 v1 不再追求“更像一个完整系统”，而要先把边界压实成“能启动一条真实任务、能按条件分流、能留下可复盘证据”的最小运行产品。
- 下一步最重要的不是补更多角色说明，而是把 `规划边界`、`必需/非必需边界`、`正式验收口径`、`最小真实任务闭环页` 一次写清。

**v1 规划边界**
- v1 = 一套能支持单个真实中小任务完成“发起—分发—执行—回收—验收—归档”的多 agent 协作模板仓。
- 任务边界：只覆盖单任务闭环，不覆盖多任务并发编排或长期项目群管理。
- 角色边界：只保证核心链路能跑通，核心是 `main + chengshu + tieshou`，其他 specialist 为按需接入。
- 风控边界：只要求具备“按风险触发 review/release”的能力，不要求所有任务都走完整审批链。
- 证明边界：v1 的成立证据不是“文档齐全”，而是“至少一条真实场景闭环可复现、可检查、可交接”。

**v1 必需项 / 非必需项边界清单**
- v1 必需项：
  - 核心角色最小集：`main / chengshu / tieshou`
  - 最小 runtime 对象集：`task-card / delegation-request 或等价发起物 / handoff / review / release / archive record`
  - 初始化材料：`bootstrap checklist`、`team activation sheet`、`first-task starter pack`
  - 操作规则：`runtime-sop`、`spawn-decision-guide`
  - 一条最小真实任务样例
  - 正式验收清单
  - 失败收束机制
- v1 非必需项：
  - 所有 specialist 的深度完善与高密度模板打磨
  - planner 的工程实现、自动映射脚本、复杂依赖树生成
  - 长期运营度量体系
  - 多层级 review 流程、复杂 release 分级制度
  - 重型目录重构与长周期归档治理

**DoD 转正式验收口径**
- 验收条款 A：可启动
  - 怎么检查：给未参与搭建的人一份新任务说明，看其能否仅依据初始化材料完成首轮建档与角色启用
  - 由谁检查：主 Agent 或项目维护者
  - 产物：已填写的 `task-card`、已确定的 `team activation sheet`、首轮 runtime 文件骨架
- 验收条款 B：可分发
  - 怎么检查：从任务卡出发，是否能形成清晰 delegation，并由承枢/执行位接手，不靠临时聊天补语义
  - 由谁检查：承枢 + 旁观复核者
  - 产物：`delegation-request`、执行接单记录、必要的 handoff 占位
- 验收条款 C：可收口
  - 怎么检查：执行完成后，是否有明确回收节点；任务结果是否被主链接住，而不是散落在聊天或临时文件里
  - 由谁检查：主 Agent
  - 产物：`handoff`、结果摘要、状态更新记录
- 验收条款 D：可风控
  - 怎么检查：分别抽一条低风险任务与一条高风险任务，检查是否按规则分流；高风险是否触发 review gate，低风险是否允许轻量闭环
  - 由谁检查：铁手或指定 review 角色
  - 产物：`review card`（如触发）、`release decision`
- 验收条款 E：可复盘
  - 怎么检查：由未参与该任务的第三人只看 runtime 留档，判断能否复述任务目标、分工、过程、结果
  - 由谁检查：问隙或其他未参战角色
  - 产物：归档记录、最终状态、必要的失败/偏差说明

**最小真实任务场景一页式闭环草案**
- 场景名称：已有仓库中一项规则修订，并同步更新一处关联说明文档
- 任务目标：
  - 修改一处既有规则/模板内容
  - 同步修正一处相关说明文档，保证前后一致
  - 输出一份明确结果说明，说明改了什么、为什么改、影响到哪里
- 必经节点：
  - 主 Agent 建立 `task-card`
  - 启用 `main / chengshu / tieshou`
  - 承枢完成拆解与 delegation
  - 铁手执行修改
  - 执行结果通过 `handoff` 回收
  - 主 Agent 做结果确认并归档
- 条件节点：
  - 若改动涉及高风险事项：触发 `review gate`
  - 若结果需正式对外放行：补 `release decision`
  - 若任务中途失败：触发 `fallback / rollback note`
- 结束条件：
  - 目标文件已修改完成
  - 关联说明已同步
  - handoff 已回收
  - 是否需要 review/release 已有明确判定
  - runtime 留档足够第三人复盘

**给问隙的接力重点**
- 把正式验收清单写成第三方检查视角
- 把归档口径和复盘口径分开
- 明确 v1 的“通过 / 不通过 / 有条件通过”三种验收结论分别如何落文

### 问隙
**结论**
- v1 验收不能继续站在“项目自述视角”，而要改成“第三方检查视角”。
- 当前最大的风险不是文件缺失，而是“看起来能闭环、实际上第三方不可独立核验”。
- 建议顺序：先把验收口径外部化，再拆开归档与复盘口径，再定义三档验收结论，最后补高漏点检查项。

**第三方检查视角的正式验收清单改写**
- 每条统一写成：`检查目标 / 检查动作 / 通过证据 / 不通过信号`
- 可启动：第三方按初始化材料独立执行，能否在限定时间内完成首轮建档与角色启用
- 可分发：第三方检查 delegation 是否足以让未参战执行位直接接手，不依赖额外口头补充
- 可收口：第三方是否能仅凭 runtime 留档确认任务结果、责任交接点、结束状态
- 可风控：第三方抽检高低风险样例时，是否能看到一致的触发依据、放行依据与责任人
- 可复盘：第三方是否能在不询问参与者的前提下复述任务目标、路径、偏差、结果、遗留项

**归档口径 vs 复盘口径**
- 归档口径：解决“这个任务最后要留下什么正式记录，供以后查账和接续”
  - 只收结果性事实：任务编号、目标、责任角色、最终状态、关键产物位置、是否放行、遗留事项
  - 追求：短、稳、可检索
- 复盘口径：解决“这次为什么这样做、哪里偏了、下次怎么少犯错”
  - 收解释性内容：判断是否成立、信息损失点、低效节点、模板缺口、改进动作
  - 追求：有偏差解释、有改进动作

**v1 三种验收结论的落文方式**
- 通过：核心链路已跑通，第三方可独立检查，缺陷不影响 v1 主张成立
- 不通过：存在一处或多处断裂，导致“可启动 / 可分发 / 可收口 / 可风控 / 可复盘”中至少一项不成立
- 有条件通过：主链路成立，但已知缺口已明确圈定，且该缺口不阻断当前受限使用
- 建议统一格式：`结论 + 依据 + 限制 + 后续动作`

**当前最容易漏掉的 QA / 风险点**
- 状态漂移：文件状态和真实推进状态不一致
- 条件节点漏判：本应触发 review/release 的任务被轻量放过，或反过来把低风险任务过度审批
- 交接不可执行：handoff 存在，但不足以让第三方接手
- 证据不可追溯：结果存在，但依据、放行理由或责任人不清
- 失败收束缺位：任务中断后没人收口、没人留痕、没人标明后续动作

**给观象的接力重点**
- 设计第三方检查入口
- 收一页式验收总览
- 补风险可视化方式，让“当前任务处于什么状态、风险在哪”更容易被看见

### 观象
**结论**
- v1 现在最缺的不是规则文本，而是“检查界面”。
- 第三方需要一眼看出：任务是否跑通、风险卡在哪、哪些只是文件存在但状态未成立。
- 建议顺序：先做第三方检查入口，再做一页式验收总览，再补风险/状态可视化，最后把 structured ops 接到状态维护位。

**第三方检查入口设计**
- 建议增加固定入口页：`runtime/check-entry.md` 或 `runtime/acceptance-overview.md`
- 入口页只回答 5 个问题：
  - 这次验收检查哪一个任务
  - 当前验收结论是什么
  - 检查人先看哪几个文件
  - 每个文件对应验证什么
  - 当前已知限制与未通过点是什么
- 建议第三方检查顺序固定成：
  - 先看 acceptance-overview
  - 再看 task-card
  - 再看 team activation sheet
  - 再看 delegation / handoff
  - 再看 review / release
  - 最后看 archive / recap

**一页式验收总览 / 看板思路**
- 单页只放四块信息：
  - 任务摘要
  - 状态总览
  - 风险总览
  - 结论区
- 建议状态色：
  - 绿：已完成且证据齐
  - 黄：进行中或有条件成立
  - 红：阻塞、漏判、证据缺失
  - 灰：当前不适用
- 核心要求：15 秒能定位问题，而不是让检查人自己拼接多份文件。

**风险可视化与状态可视化**
- 每个关键节点不仅要有 `status`，还要有 `evidence`。
- 例如：
  - `delegated = yes` 还要补 `evidence = delegation-request-xxx.md`
  - `review_required = no` 还要补 `reason = low-risk rule matched`
  - `done = yes` 还要补 `evidence = handoff + archive record`
- 建议重点高亮的异常：
  - 有文件但无证据链接
  - 有 handoff 但无接收人
  - 任务标记完成但无 review/release 判定
  - 归档已写但 runtime 主状态仍未结束

**structured ops 最适合介入的点**
- 状态归一
- 证据挂接
- 验收视图生成
- 异常扫描
- 也就是把“判断结果”变成“可见状态”，而不是替代主 Agent 做判断。

**给片场的接力重点**
- 设计最小可用的一页式验收总览 / 状态看板模板
- 设计第三方检查动线
- 定义异常状态展示规则：哪些必须标红，哪些只能判“有条件通过”

### 铁手
**结论**
- 我同意承枢的主判断：项目下一步的核心不是“再补说明”，而是把模板压缩成一套能开工、能收口的最小运行面。
- 但我建议把 P2“最小实战闭环样例”前置到和验收层并行设计，因为没有样例，验收标准很容易继续写空。
- 所以顺序我修正为：`初始化层 -> 最小闭环样例 + 验收层 -> 长期运行机制 -> 目录再收口`。

**1. 对承枢判断的同意与修正**
- 同意点：当前最大断层就是“首日怎么跑起来”。这一层不补，现有角色模板、runtime 对象、SOP 都还停在会用的人能脑补，不会用的人无法开跑。
- 同意点：现在不该继续横向扩资料，应该转向首轮真实运行闭环。
- 修正点：验收层不能单独长在文档里，必须绑定一个最小案例。不然“验收通过”只是在检查文件是否存在，不是在检查协作是否成立。
- 修正点：`review gate / release decision` 不应一刀切视为每个任务必经，而应做成“按风险触发”的条件节点；否则 v1 的启动成本会过高。

**2. 我认为初始化层和验收层应具体长成什么样**
- 初始化层要解决的不是“目录生成”，而是“第一次接任务时谁先做什么”。
- 初始化层建议最少包含 4 类产物：
- `workspace bootstrap checklist`：首日建哪些目录、复制哪些模板、哪些字段必须改成真实值。
- `team activation sheet`：本次启用哪些角色，谁常驻，谁按需唤起，避免 8 个角色默认全开。
- `first-task starter pack`：第一次任务至少要生成哪些对象，建议固定为 `task-card + delegation-request + handoff/review 占位节点`。
- `fallback / rollback note`：首次运行失败时怎么收束，至少规定“谁负责回收现场、如何记录失败原因、哪些文件必须补写”。
- 初始化后的最小目录，不该追求完整，而应追求够跑：
- 必需：`team/main`、`team/chengshu`、`team/tieshou`、`runtime/task-card.*`、`runtime/handoff.*`、`runtime/review.*`、`runtime/release.*`、`examples/` 中一条可照抄的闭环样例。
- 可按需启用：`tanzhen / biguan / wenxi / guanxiang / pianchang` 等 specialist 目录和对应模板。
- 验收层建议也拆成 4 个明确检查面：
- `启动验收`：能否基于模板在 30 分钟内拉起一次任务，不靠额外口头解释。
- `流转验收`：任务是否真的经过主 Agent -> 承枢/执行 -> 交接/回收，而不是只填静态文件。
- `风控验收`：高风险任务是否能触发 review gate，低风险任务是否允许轻量放行。
- `归档验收`：任务结束后是否留下可复盘材料，下一次是否能基于记录继续。
- v1 的验收标准应尽量定成“观察得到的行为”，不要定成“文件数量达标”。

**3. 当前模板里哪些是 v1 必需，哪些可暂缓**
- v1 必需：
- `main / chengshu / tieshou` 三个核心角色模板。因为这是最小指挥、拆解、执行链，没有这三者就不构成团队协作。
- `task card / handoff / review / release` 四类 runtime 对象。即便部分是轻量使用，也要先把对象关系定住。
- `runtime-sop` 与 `spawn-decision-guide`。前者解决怎么跑，后者解决什么时候不用乱起 agent。
- 至少一套完整闭环样例，而且这个样例要覆盖：发起、分发、执行、回收、验收、结束。
- 初始化与验收的清单型文档。这是当前最缺的落地层。
- v1 可暂缓：
- 全角色高密度打磨。现阶段 specialist 写得再漂亮，也不如先证明最小链路能稳定跑。
- planning contract 的深扩与 planner 映射细化。现有 contract 先够占位，别在本体阶段继续做重设计。
- 长任务归档策略、历史任务分析、度量指标体系。这些重要，但属于跑起来之后再收敛。
- 更复杂的 release 分级、review 多层机制、目录再次重构。v1 先有条件分流即可，不必制度过满。

**4. 我建议下一位 agent（探针）重点接什么问题**
- 请探针不要再泛谈优先级，而是直接回答一个问题：`什么叫“这个模板仓已经真的能跑 v1 了”？`
- 重点建议他补三件事：
- 第一，定义一个“最小真实任务场景”，例如一次小型文档修订、配置修复或内容交付，要求能完整走完链路。
- 第二，明确这个场景下哪些节点是必经、哪些节点按风险触发，特别是 review / release 的判定条件。
- 第三，给出一份 v1 完成定义（DoD），必须是可检查、可观察、可复用的，不要再写成抽象原则。

**一句话判断**
- 这个项目本体现在最该补的不是更多能力，而是“最小开跑包 + 最小验收包 + 一条真实闭环样例”，先证明它是操作系统，不只是模板集合。

### 问隙
**结论**
- 我同意前面几位的收敛方向，但还要再补一个约束：v1 验收不能继续站在“项目自述视角”，而要改成“第三方检查视角”。
- 原因很简单：现在最容易出现的假通过，不是文件缺失，而是“写得像闭环、实际上不可核验”。
- 所以下一步建议顺序是：`先把验收口径外部化 -> 再把归档与复盘拆口径 -> 再定义三档验收结论写法 -> 最后补最容易漏的 QA 检查点`。

**1. 如何把正式验收清单改写成第三方检查视角**
- 核心改法：每一条都不要写“项目方做了什么”，而要写“第三方检查人能看到什么、如何判定、缺什么算不过”。
- 我建议把正式验收清单统一改成 4 列：`检查目标 / 检查动作 / 通过证据 / 不通过信号`。
- 示例落法：
- `可启动` 不写“已提供 bootstrap checklist”，而写“第三方按 bootstrap checklist 独立执行，能否在限定时间内完成首轮建档与角色启用”。
- `可分发` 不写“已有 delegation 模板”，而写“第三方检查 delegation 文本是否足以让未参战执行位直接接手，不依赖额外口头补充”。
- `可收口` 不写“已生成 handoff”，而写“第三方能否仅凭 runtime 留档确认任务结果、责任交接点、结束状态”。
- `可风控` 不写“支持 review gate”，而写“第三方抽检高低风险样例时，是否能看到一致的触发依据、放行依据与责任人”。
- `可复盘` 不写“已完成归档”，而写“第三方是否能在不询问参与者的前提下复述：任务目标、路径、偏差、结果、遗留项”。
- 判断上要增加反证：凡是必须靠聊天记录、口头说明、个人记忆补齐的，一律不算通过证据。

**2. 如何区分“归档口径”和“复盘口径”**
- 我的判断：这两个经常被混写，但作用完全不同，必须拆开。
- `归档口径` 解决的是“这个任务最后落什么正式记录，供以后查账和接续”。
- `复盘口径` 解决的是“这次为什么这样做、哪里偏了、下次怎么少犯错”。
- 所以建议明确分工：
- `归档口径` 只收结果性事实：任务编号、目标、责任角色、最终状态、关键产物位置、是否放行、遗留事项。
- `复盘口径` 才收解释性内容：原判断是否成立、哪里出现信息损失、哪个节点效率低、哪个模板字段不够用、下轮怎么改。
- 归档追求 `短、稳、可检索`，是面对未来接手人和管理者的。
- 复盘追求 `有偏差解释、有改进动作`，是面对团队内部迭代的。
- 不能把复盘情绪、争论过程、未证实猜测塞进归档；也不能只留一条“已完成”就当复盘结束。

**3. v1 的“通过 / 不通过 / 有条件通过”三种验收结论应如何落文**
- 建议不要只写结论词，必须固定成“结论 + 依据 + 限制 + 后续动作”的短格式，不然结论无法执行。
- `通过` 的落文标准：核心链路已跑通，第三方可独立检查，缺陷不影响 v1 主张成立。
- 建议句式：`结论：通过。依据：已完成 X、Y、Z 检查，第三方可按留档复述闭环。备注：存在若干优化项，但不影响 v1 作为最小运行模板投入使用。`
- `不通过` 的落文标准：存在一处或多处断裂，导致“可启动 / 可分发 / 可收口 / 可风控 / 可复盘”中至少一项不成立。
- 建议句式：`结论：不通过。原因：当前在 X 节点仍需依赖口头补充/人工兜底，第三方无法独立完成检查。处置：补齐 A、B 后重新验收。`
- `有条件通过` 的落文标准：主链路成立，但已知缺口被明确圈定，且该缺口不阻断当前受限使用。
- 建议句式：`结论：有条件通过。边界：当前仅适用于 X 类低/中风险单任务；涉及 Y 情况时不得直接放行。待补：需在 N 项补齐后转正式通过。`
- 我的建议是：v1 很可能首次验收会落在“有条件通过”，这不是坏事；比“形式上直接通过、后续靠人补洞”更稳。

**4. 当前方案里最容易漏掉的 QA / 风险点**
- 我认为最容易漏的不是模板缺页，而是“责任与状态的可验证性”。
- 具体有 5 个高漏点：
- `状态漂移`：task-card、handoff、release 上的状态不一致，最后没人说得清任务到底结束没有。
- `条件节点误触发或漏触发`：review / release 写成“可选”，结果高风险任务也被当普通任务直接收口。
- `交接文本字段齐了但不可执行`：表面有 handoff，实际没有边界、输入、完成条件，接手人仍要追问。
- `结果存在但证据不可追溯`：改动做了，但 runtime 留档里找不到对应产物路径、版本点或最终确认人。
- `失败收束缺位`：任务没做成时没有 rollback note、没有失败原因、没有现场回收责任，导致下次重复踩坑。
- 如果只补文档不做这些检查，v1 最后最容易变成“有模板的手工流程”，而不是“可审计的协作系统”。

**5. 建议下一位 agent（观象）重点接什么问题**
- 我建议观象不要继续讲文字结构，而要专门解决：`第三方拿到这套仓库时，如何一眼看懂“现在能不能用、适用于什么边界、风险卡在哪里”`。
- 具体请他重点补 3 件事：
- 第一，把 `通过 / 有条件通过 / 不通过` 三档结论做成统一展示面，最好是一页式总览，不要藏在长文里。
- 第二，设计“第三方检查入口顺序”，也就是检查人先看哪几个文件就能完成快速判断。
- 第三，补“风险可视化”方案：哪些节点是必经，哪些是条件触发，哪些状态说明任务仍不可放行。

**一句话判断**
- 这个项目本体现在最该补的，是把“自我说明”改成“第三方可核验”，否则 v1 很容易在纸面上成立，在检查时失真。

## Round 2 统一整理

### 第二轮整体判断
第二轮的讨论已经足够收敛，不再需要为项目本体单独开启第三轮。核心原因不是“大家说得差不多”，而是 5 个未决点里，真正影响 v1 收口的关键项已经形成稳定共识，剩余差异主要落在写法和展示层细节，适合由主 Agent 直接裁决并转成正式条文。

### 第二轮已形成的稳定共识

**1. 唯一验收样例**
- 正式定为：`已有仓库中的单文件规则/模板修订，并同步更新一处关联说明文档`。
- 采用原因：
  - 真实，而不是演示性假任务
  - 足够窄，不会把 v1 拖进开发、测试、发布等重链路
  - 足够完整，能覆盖 `main -> chengshu -> tieshou -> 回收/验收/归档` 主链
  - 易于第三方检查差异、一致性、证据链和条件节点判定
- 不采用：
  - `纯新写文档`：太轻，难以证明协作链与一致性控制成立
  - `开发 + 测试 + 发布`：太重，会把首次验收噪音放大，削弱 v1 边界

**2. 初始化包的最小正式范围**
- 主裁决：初始化包正式范围按 **4 件** 收口：
  - `bootstrap checklist`
  - `team activation sheet`
  - `first-task starter pack`
  - `fallback / rollback note`
- 其中 `first-task starter pack` 的最小对象固定为：
  - `task-card`
  - `delegation-request`
  - `handoff` 占位
  - `review / release` 判定占位
- 裁决理由：
  - 初始化包的目标是“第一次能开跑”，不是“把所有对象一次铺满”
  - 把 `acceptance-overview` 放进初始化包会让“起跑包”和“检查面”混在一起，职责不够清晰
- 补充裁决：
  - `acceptance-overview` 不计入初始化包，但计入 **v1 必备检查面**，应在首个真实任务运行中同步生成

**3. `archive` 与 `recap` 是否分对象**
- 正式定为：**从 v1 就分对象**。
- 分工如下：
  - `archive`：只留结果性事实，面向查账、检索、接续
  - `recap`：只留复盘性判断，面向纠偏、改进、模板迭代
- 裁决理由：
  - 不分对象会让正式归档被过程解释污染
  - 也会让复盘为了保持简短而失真
  - 第三方检查时，混合对象不利于快速判断“任务到底完成了什么”
- 轻量化边界：
  - v1 的 `recap` 可以轻，不要求长文，但对象边界必须现在就分开

**4. 状态维护的人机分工**
- 正式定为：**人负责判定，结构化层负责落账、挂证据、扫异常、生成总览，不替代人做裁决。**
- 责任划线：
  - `main`：任务总状态、是否通过、是否归档、是否进入下一阶段
  - `chengshu`：流转状态、分发状态、是否进入 review/release 判定区
  - 执行位：提交执行结果、说明完成边界、交回 handoff
  - structured ops / 检查视图：统一状态字段、链接证据、扫描冲突、生成 overview
- 强规则：
  - 任何 `完成 / 通过 / 已归档` 状态都必须同时给出 `evidence` 或 `reason`
  - 系统可以提示“缺证据 / 状态冲突 / 条件节点漏判”，但不能自动宣布“通过”

**5. 首次验收目标档位**
- 正式定为：**首次验收目标 = 有条件通过**。
- 含义：
  - 不是降低标准，而是承认当前阶段仍处于“主链路成立验证”阶段
  - 首次验收证明的是：`主链路成立 + 第三方可核验 + 边界已写清`
  - 不是证明“该体系已适用于所有任务类型”
- 首次“有条件通过”的适用边界：
  - 单任务
  - 低/中风险
  - 核心链路 `main + chengshu + tieshou`
- 暂不纳入首次正式通过范围：
  - 高风险任务
  - 重型开发/测试/发布链
  - 长周期任务
  - 多任务并发编排

### 第二轮剩余差异与主 Agent 裁决
第二轮真正存在的差异，主要在两个点：

**差异 1：`acceptance-overview` 是否算初始化包正式成员**
