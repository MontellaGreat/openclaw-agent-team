# taolun2

## 讨论主题
围绕“智能任务规划系统”专项，进行独立多 Agent 讨论，目标是形成专项计划书，并明确它如何作为项目本体的外部规划能力接入，而不替代主治理链。

## 项目参考基线
本专项讨论以 GitHub 项目 `2025Emma/vibe-coding-cn` 为方法论参考基线，重点吸收其中这些思想：
- 规划优先，而不是先写后补
- 上下文优先，垃圾进垃圾出
- 文档即上下文，不是事后补充
- 接口先行，实现后补
- 一次只做一件事，避免 AI 失控扩写
- AI 负责辅助与执行，不直接越权接管治理

## 已定前提
- 项目本体 v1 已完成两轮讨论并收口到正式规划条文
- 智能任务规划系统不替代治理层
- planner 输出必须经过主 Agent 审核后才能进入主链
- 当前阶段优先讨论：它解决什么、边界是什么、怎么接入、怎么验收
- 本专项应设计成“外部规划层”，而不是“自动总控层”

## 专项定位
一句话定位：
`智能任务规划系统 = 一个把“目标 → 上下文 → 任务拆解 → 执行顺序 → 风险边界 → 验收口径”结构化输出的外部规划层。`

它服务于主 Agent，不替代主 Agent。

## 新增讨论前提（本轮并入）
在原有“外部规划层”定位不变的前提下，本专项新增纳入以下需求：
- 需要提供一个 `shell` 命令入口，允许人在后台直接下发任务
- 需要支持由主 Agent 主动发起任务，而不是只接受人工输入
- 任务布置下去后，系统不能只负责拆解，还必须具备持续盯进度、轮次推进、超时检查、失败重试的能力
- 系统应支持“任务一直推进直到完成 / 阻塞 / 人工终止”，而不是单次规划后即退出
- 该能力仍属于“调度与执行监督增强”，不等于取得治理裁决权

## 专项升级定位
结合新增需求，本专项从“纯规划器”升级为：

`智能任务规划系统 = 一个由 shell / 主 Agent 双入口驱动，负责任务整形、拆解、派发、轮次推进、状态跟踪、失败重试，并在治理边界内持续监督任务执行的任务编排系统。`

注意：
- 它可以持续推进任务
- 但它不能替代主 Agent 做高风险裁决
- 它是“编排与监督内核”，不是“自动总控”

## 讨论范围
本专项只讨论：
1. 智能任务规划系统的目标与非目标
2. 输入 / 输出 contract
3. 与主 Agent / 承枢 / runtime 的边界关系
4. 介入时机、退出时机与否决机制
5. 风险、失败模式、验收方法
6. 如何基于 `vibe-coding-cn` 的方法论把规划系统做成“可控的前置规划层”
7. 如何把它扩展成“可持续推进的任务编排系统”而不越权
8. 如何增加 shell 命令入口、主 Agent 发起入口与后台进度监督能力

## 讨论原则
1. 不重复项目本体已收敛的内容
2. 不把 planner 写成主裁决者
3. 不以“更智能”为理由扩大系统边界
4. 优先讨论可接入、可控、可验收，而不是先讨论炫技能力
5. 输出必须尽量能映射到 runtime 对象，而不是停留在抽象建议层
6. 新增能力优先落到“统一入口、任务状态、推进机制、监督机制”，而不是空泛描述“自动化”
7. 任何持续推进、自动重试、超时催办能力，都必须有明确停止条件与人工接管条件

## 第二轮必须纳入的方案稿

### 一、目标
本专项第二轮应把系统目标从“生成计划”进一步收敛为以下四件事：
- 把来自人或主 Agent 的任务统一收口到同一任务总线
- 把任务转成可跟踪、可派发、可推进、可中断的状态对象
- 在后台持续监督任务进度，按轮次或事件推进后续动作
- 在不越权的前提下，把阻塞、失败、超时和高风险步骤稳定抬回主 Agent

### 二、系统边界
它应该做：
- 接任务
- 整理上下文
- 拆解执行步骤
- 建立依赖关系
- 生成任务状态对象
- 派发给附属 agent / 外部执行单元 / 人工待办
- 记录进度
- 到点检查
- 失败重试
- 识别阻塞
- 在必要时回收给主 Agent 审议

它不应该做：
- 直接替代主 Agent 做最终裁决
- 自动放行高风险动作
- 未经审议直接改配置、发外部消息、做正式发布
- 为了“持续推进”无限循环、自主扩权、自主改目标

### 三、统一入口设计
建议系统至少有两类入口：

1. `shell` 入口
- 示例：`openclaw task create "整理并发布温州项目合作方案"`
- 作用：允许用户本人在后台直接发任务
- 要求：支持纯文本快速下发，也支持后续补充结构化参数

2. 主 Agent 发起入口
- 由主 Agent 在判断“值得派发 / 值得进入编排”后主动创建任务
- 作用：把“主 Agent 的判断”接到“任务编排系统”的执行链上
- 要求：保留主 Agent 的审批权与叫停权

后续可扩展但本轮不强求：
- Feishu / IM 入口
- Web 面板入口
- API 入口

### 四、最小任务对象
系统至少应维护一个统一的任务对象，建议最小字段如下：
- `task_id`
- `source`（shell / main-agent / 其他）
- `creator`
- `goal`
- `context`
- `constraints`
- `priority`
- `status`
- `current_round`
- `steps`
- `depends_on`
- `assignee`
- `last_update_at`
- `next_check_at`
- `retry_count`
- `risk_level`
- `approval_required`
- `done_definition`
- `result_summary`
- `blocked_reason`

### 五、最小状态机
建议最小状态集：
- `queued`
- `planning`
- `ready`
- `running`
- `waiting`
- `blocked`
- `retrying`
- `review_required`
- `done`
- `failed`
- `cancelled`

状态要求：
- 任何状态迁移都应有触发原因
- `blocked`、`review_required`、`failed` 必须可追溯原因
- `retrying` 必须有次数上限和升级条件
- 长时间无更新的 `running` / `waiting` 任务必须可被监督器识别

### 六、推进机制
第二轮必须讨论清楚，系统不是“一次性出计划”，而是按以下机制持续推进：
- 有新任务进入时，先整形输入并生成计划草案
- 可执行步骤进入派发
- 派发后进入等待或运行态
- 到 `next_check_at` 时由监督器复查
- 若步骤完成，则推进下一轮可执行步骤
- 若超时无反馈，则进入催办 / 标记阻塞 / 升级主 Agent
- 若失败且可重试，则进入 `retrying`
- 若触及高风险边界，则进入 `review_required`
- 若达到完成定义，则收口为 `done`

### 七、监督器职责
后台监督器至少应负责：
- 定时扫任务
- 找出超时未更新任务
- 找出依赖已满足但未推进任务
- 找出重试次数过高任务
- 找出长时间卡住的 `blocked` 任务
- 把高风险、冲突、异常回收给主 Agent

监督器不是裁决者，它是：
- 进度盯梢者
- 异常发现者
- 轮次推进触发器
- 升级上报器

### 八、与主 Agent 的关系
主 Agent 负责：
- 判断任务是否值得进入编排系统
- 审核高风险步骤
- 处理异常升级
- 审核最终对外放行项

任务规划系统负责：
- 接任务
- 编排
- 跟踪
- 推进
- 监督
- 升级

结论：
- 主 Agent 是治理中心
- 任务规划系统是执行编排中枢
- 两者是“治理 + 编排”关系，不是替代关系

### 九、第二轮重点问题
Round 2 建议重点回答以下问题：
1. `shell` 入口最小命令集怎么定义
2. 主 Agent 发起任务时，最小 contract 是什么
3. 任务状态对象最小字段集是否足够
4. 监督器按“轮询”还是“事件 + 轮询混合”推进更稳
5. 什么情况自动重试，什么情况必须升级主 Agent
6. 什么叫“任务还活着”，什么叫“任务已经卡死”
7. 如何防止任务编排系统无限推进、无限催办、无限重试
8. 第二轮结束后，能否产出一版 runtime 可映射的接口草案

## 需要重点回答的 6 个主题

### 主题 1：输入层
- 规划系统的最小输入是什么
- 输入如何分层：目标 / 非目标 / 上下文 / 约束 / 当前状态 / 完成定义
- 哪些输入必须结构化，哪些可保留自由文本

### 主题 2：输出层
- 规划结果应该输出成什么对象
- 最小输出字段是什么

### 主题 3：与主 Agent 的边界
- planner 什么时候可以介入

- 哪些内容 planner 只能建议、不能定案

### 主题 4：规划粒度控制
- 哪些任务值得启 planner
- 哪些任务应直接做，不值得重规划
- 如何避免过度拆解、过度规划、过度生成

### 主题 5：失败模式
- 上下文不足时是否应拒绝规划
- 什么叫“看起来完整但不可执行”
- 如何做失败回退与否决

### 主题 6：专项验收
- 这个规划系统如何证明自己有效
- 首次验收用什么样例
- 什么叫“计划可执行”
- 如何定义通过 / 有条件通过 / 不通过

## 第二轮新增参考材料：Subagent 自主项目管理
本轮新增参考资产：

### 可吸收的部分
这份材料中，有 5 个点值得吸收进入第二轮：

1. 共享状态文件思路
- 用统一状态文件作为单一事实来源，避免任务进度只散落在消息里
- 这与本专项里的“任务状态对象 / 任务账本 / 监督器”是兼容的
- 第二轮可以讨论：是否采用 `STATE.yaml`、`tasks.yaml`、`jsonl` 或其他持久化形式

2. 标签化 PM / 子任务负责人模式
- 通过稳定的 label 或 owner 约定，把一个持续性任务交给固定的子 agent 负责
- 这有利于长期任务的连续推进与责任归属
- 第二轮可以讨论：`pm-{project}-{scope}` 这类命名是否适合作为任务编排系统的实例约定

3. 主 Agent 精简、子 Agent 执行
- 主 Agent 不必参与所有细节执行，而应聚焦判断、派发、复核、收口
- 这个思路与本专项“主 Agent 是治理中心，规划系统是执行编排中枢”的定位一致

4. 并行工作流管理
- 复杂项目往往不是单链任务，而是多条并行工作流
- 第二轮应讨论：任务编排系统如何表示并行子任务、共享依赖、汇总收口

5. 状态持久化 + 历史可追溯
- 如果状态变更能被持久化，并最好有历史记录，就更适合长期任务和中断恢复
- 这与本专项“持续推进而不是单轮即忘”高度一致

### 不直接照搬的部分
这份材料里，也有几点不能直接照搬，第二轮必须显式讨论清楚：

1. 不能完全去中心化
- 该材料强调“去中心化、弱中央调度”
- 但本专项已经明确：高风险动作、正式放行、异常升级，仍必须回到主 Agent
- 所以它最多只能作为“分布式执行参考”，不能变成“无治理中心”

2. 不能把共享状态文件等同于全部系统能力
- `STATE.yaml` 只是状态载体，不等于完整的任务编排系统
- 本专项还需要入口、状态机、监督器、重试、升级、停止条件这些能力

3. 不能默认主会话只做 0-2 次调用
- 该材料倾向于把主会话压到极简
- 但本专项里，主 Agent 仍要承担判断、审议、升级处理、风险裁决，因此不能被约束成几乎不干预

4. 不能默认所有状态都交给 Git 提交
- Git 审计是可选增强，不一定是最小实现的唯一依赖
- 第二轮可以讨论“是否把 Git 作为状态审计增强层”，但不应在最小方案里强绑定

### 第二轮新增讨论点
基于该参考材料，第二轮可新增以下议题：
- 任务编排系统是否应引入共享状态文件机制
- 共享状态文件与任务状态机如何映射
- 一个持续项目是否应由固定 `PM 子 Agent` 持有并维护状态
- 主 Agent、PM 子 Agent、执行子 Agent 三者如何分层
- 状态文件、消息调度、监督器轮询三者如何配合
- 如何避免“去中心化执行”演变成“无人兜底的自治系统”

### 当前建议立场
本专项可以吸收该材料的这条核心思想：

`让子 Agent 围绕共享状态持续推进项目，而不是每一轮都靠主 Agent 重新手工协调。`

但必须同时保留这条底线：

`共享状态提高执行自治，不等于取消主 Agent 的治理权、审批权与收口权。`

## Round 2 收敛结果（主 Agent 汇总）
第二轮讨论已形成较强共识。本轮不再把“智能任务规划系统”理解为单次输出计划的外部规划器，而是进一步收敛为：

`一个由 shell 与主 Agent 双入口驱动，具备任务编排、状态持久化、持续推进、异常监督、有限自动重试与升级回主 Agent 机制的任务编排系统。`

### Round 2 已形成的核心共识

#### 1. 系统定位共识
- 系统不是“自动总控”
- 系统不是“去中心化治理替代品”
- 系统是“执行编排中枢 + 监督内核”
- 主 Agent 的治理中心地位不变
- 高风险动作、正式放行、异常升级、目标改写，必须回到主 Agent

一句话收敛：

`主 Agent 负责治理，任务编排系统负责持续推进。`

#### 2. 结构共识
第二轮综合意见后，系统结构可收敛为四层：

1. `入口层`
- shell 命令入口
- 主 Agent 发起入口

2. `编排层`
- 输入整形
- 任务拆解
- 依赖建模
- 派发执行
- 轮次推进

3. `监督层`
- 事件触发推进
- 轮询兜底检查
- 超时识别
- 阻塞识别
- 异常升级

4. `状态持久化层`
- 任务当前状态快照
- 历史事件日志
- 可恢复、可追溯、可审计

其上始终覆盖一条治理边界：
- 主 Agent 掌握准入、审议、升级处理、最终收口权

#### 3. 入口共识
系统最小必须具备两类入口：

1. `shell` 入口
- 允许用户本人在后台直接下发任务
- 支持纯文本快速创建任务
- 后续可补充结构化参数

2. 主 Agent 入口
- 主 Agent 在判断“值得进入编排系统”后发起任务
- 编排系统接收的是清晰 contract，而不是模糊意图

后续可扩展入口：
- Feishu / IM
- API
- Web 面板

但这些都不是第二轮的最小闭环必需项。

#### 4. shell 最小命令集共识
多份意见对 shell 命令集略有差异，但可收敛到以下最小集合：

- `openclaw task create "<goal>"`
- `openclaw task list`
- `openclaw task show <task_id>`
- `openclaw task cancel <task_id>`
- `openclaw task retry <task_id>`

其中：
- `create` 是创建入口
- `list/show` 是观察入口
- `cancel` 是人工停止入口
- `retry` 是人工恢复入口

结论：
- 第二轮不建议一开始就扩出 `pause/resume/approve/reassign/watch/logs` 等重型命令
- 最小命令集应优先保障“创建、查看、停止、恢复”闭环

#### 5. 主 Agent 发起 contract 共识
主 Agent 发起任务时，至少应给出一份最小委托 contract。

多份意见收敛后，建议最小必需字段如下：
- `goal`
- `context`
- `constraints`
- `done_definition`
- `priority`
- `risk_level`
- `approval_required`
- `source`
- `creator`

结论：
- 编排系统不能靠猜测理解任务
- `done_definition` 必须可验证
- `risk_level` 与 `approval_required` 共同定义治理边界
- `steps`、`status`、`retry_count` 这类运行态字段不应由入口 contract 预先指定

#### 6. 状态对象共识
当前文档中列出的最小任务字段集基本够用，但第二轮一致认为还需小幅补强。

建议补充的关键字段包括：
- `created_at`
- `updated_by`
- `review_reason`
- `terminal_reason`
- `state_file_path`
- `escalation_count`
- `history_log`（或等价历史记录机制）

结论：
- 不需要推翻现有字段设计
- 但必须补足“创建时间、最后更新责任人、进入审议原因、终态原因、状态文件位置、升级次数、历史追溯”这些运行时关键字段

#### 7. 监督器机制共识
这是第二轮最强共识之一：

`监督器采用“事件驱动为主，轮询兜底”的混合模式。`

事件触发场景：
- 新任务创建
- 子步骤完成
- 子步骤失败
- 审议结果返回
- 人工 retry / cancel
- 依赖解除

轮询兜底场景：
- `next_check_at` 已到但未推进
- `running` / `waiting` 长时间无更新
- `blocked` 停留过久
- `retrying` 超限
- 依赖已满足但未触发后续推进

边界要求：
- 监督器只负责检查、触发、标记异常、升级主 Agent
- 监督器不负责裁决
- 监督器不能改目标、补上下文、自动放行高风险动作

#### 8. 自动重试与升级边界共识
第二轮一致认为，自动重试必须是有限、低风险、可恢复的。

允许自动重试的情况：
- 瞬时性失败
- 幂等步骤失败
- 低风险动作
- 上下文未变化
- 重试预算未耗尽

必须升级主 Agent 的情况：
- 高风险动作
- 权限 / 配置 / 发布 / 外发 / 删除覆盖相关动作
- 失败原因不确定
- 上下文冲突或缺失
- 连续重试仍失败
- 长时间卡死
- 目标漂移或验收口径变化
- 需要价值判断、边界判断、方案裁决的情况

结论：
- 可自动恢复的问题自动重试
- 需要判断的问题一律升级

#### 9. 防止无限推进的共识
第二轮普遍强调，本系统必须内建止损机制，否则会从“持续推进”滑向“无限循环”。

建议纳入的硬限制包括：
- `max_rounds`
- `max_retries`
- `max_wait_duration`
- `nudge_limit`
- `task_ttl` / 截止时间
- 无进展检测
- 状态迁移白名单

结论：
- 任务必须有停止条件
- 任务必须有失败条件
- 任务必须有升级条件
- 监督器必须能识别“假推进”与“卡死”

#### 10. 对参考材料“Subagent自主项目管理”的吸收结论
第二轮确认可以吸收其以下机制：
- 共享状态文件
- PM 子 Agent 模式
- 并行协作思想
- 状态持久化与中断恢复

但不能照搬其以下治理假设：
- 不能完全去中心化
- 不能弱化主 Agent 到几乎不干预
- 不能把共享状态文件当作完整系统
- 不能默认 Git 是唯一最小依赖

本轮收敛后的立场是：

`吸收共享状态与自主推进机制，但不放弃主 Agent 的治理权、审批权与收口权。`

- 并行失控
- 升级失焦
- 交接不可接手

### 观象：默认阈值、人工接管流程、状态迁移白名单必须写死
观象第三轮最重要的补充有 5 类：

1. 共享状态账本最终规范
- 建议明确快照 + 事件日志
- 建议定义版本号、checksum、写入规则

2. 人工接管完整流程
- 监督器发现升级条件 → 标记 `review_required`
- 主 Agent 读取状态文件和日志 → 决定接管范围
- 主 Agent 更新 `takeover_plan` / `review_result`
- 监督器看到状态恢复后继续推进

3. 并行协作汇总收口机制
- 并行子任务必须有 `parent_task_id`
- 必须有汇总节点
- 汇总后父任务才允许进入下一阶段

4. 监督器自监控机制
- 监督器必须写心跳
- 心跳超时必须告警
- 主 Agent 可根据心跳判断监督器是否失效

5. 状态迁移白名单
- 合法迁移要列明
- 非法迁移要直接拒绝

观象特别强调：
- 第三轮不应只讲原则，而要把默认阈值表真正写出来
- 轮询间隔、超时阈值、blocked 超时、TTL、并发上限、催办上限等都需要明确默认值

### 片场：并行、交接、阶段收口必须制度化
片场第三轮重点补的是“产线级机制”缺口：
- 并行不等于无限派发
- 子任务并行后必须有汇总节点
- 节奏推进、交接质量、阶段收口与汇报方式都必须进一步写清

片场这一轮最关键的提醒是：
- 如果没有明确的汇总节点、交接格式、阶段收口机制
- 系统表面上会一直推进
- 但在中后段一定会因为无法汇总、无法交接、无法阶段收口而卡死

### 探针：写入协议、监督器健康检查、验收可验证性要补成主线
探针第三轮最核心的补充是 4 个工程级缺口：

1. 共享状态账本的写入协议
- 谁有写权限
- 并发写如何避免冲突
- 快照与事件日志的一致性如何保证

2. 监督器自监控 / 健康检查
- 监督器必须有心跳
- 主 Agent 或其他机制必须能检测监督器静默失效
- 监督器重启后必须能恢复未完成检查

3. 并行任务收口协议
- 并行子任务结束后由谁负责汇总
- 部分失败时如何决定：继续 / 阻塞 / 升级
- 汇总逻辑必须在任务创建时明确

4. 验收可验证性标准
- `done_definition` 不能只是抽象文字
- 必须至少包含可自动检查的条件，或明确人工验收清单

探针还补了一个非常关键的概念区分：
- `自动重试` 不等于 `自动恢复`
- 若任务已部分执行，不能简单重试，而应显式进入阻塞 / 升级 / 恢复逻辑

### 问隙：系统最缺的是可观察性、熔断、审计追溯、资源管控
问隙第三轮从风控与测试角度补得最硬：

其判断是：
- 当前设计更关注“应该做什么”
- 但严重缺乏“不能做什么”“何时必须停止”“如何发现异常”的量化机制

问隙建议必须补的 4 类硬规则：

1. 监督器自监控要求
- 心跳丢失告警
- 轮询摘要日志

2. 资源限制与熔断要求
- 任务级资源预算
- 系统级软熔断
- 资源硬上限后的强制升级

3. 审计与状态追溯要求
- 每次状态变更记录：时间 / 操作主体 / 前值 / 后值 / 理由
- 高风险动作审批链
- 回滚标记点

4. 测试与验收要求
- 验收用例库
- 边界测试点
- 失败注入测试

问隙特别强调：
- 如果这些逆向机制不补上，系统就会变成“运行时黑盒”
- 失败不会被有效约束，最终演变成“失败膨胀系统”
